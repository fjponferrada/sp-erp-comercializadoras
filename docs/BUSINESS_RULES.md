# 🧠 CORE BUSINESS RULES & AI GUIDELINES
**IMPORTANTE PARA LA IA**: Este documento es la fuente de la verdad para el comportamiento del negocio en este ERP. Antes de crear, modificar o refactorizar cualquier módulo, DEBES leer estas reglas para no romper flujos de trabajo críticos aprobados por el cliente.

## 1. Arquitectura SaaS Multi-Comercializadora y Multi-Marca
- **Sistema SaaS**: Este software está diseñado como un SaaS para ofrecer a múltiples comercializadoras eléctricas.
- **Primer Cliente (Seed Data)**: Se debe tener en cuenta que el primer cliente será la comercializadora **"aed energía"** (código `AED`), que inicialmente solo tendrá una marca llamada también **"aed energía"** (código `AED`). Habrá que importar/sincronizar sus datos.
- **Estructura Comercializadora -> Marcas**: Cada comercializadora puede dar de alta una o varias marcas bajo su mismo CIF. Cada marca tiene su propio nombre, logo y colores corporativos (para dar apariencia de desarrollo a medida).
- **Personalización Visual**: Desde la sección de Configuración, el Superadmin puede ajustar los colores y logo corporativo para cada marca.
- **Códigos Únicos**: El código de marca debe ser único entre todas las marcas. El código de comercializadora debe ser único entre comercializadoras. Una Marca sí puede compartir código con su Comercializadora padre (Ejemplo exacto: Marca "aed energía" = Código `AED`; Comercializadora "aed energía" = Código `AED`).
- **Estado de Desarrollo de la Plataforma**: 
  - EL CRM YA TIENE DESARROLLADAS LAS SIGUIENTES PANTALLAS: `Mis Comercializadoras`, `Mis Marcas`, `Incidencias`, `Dashboard`, `Análisis Económico`, `Leads`, `Contratos`, `Clientes`, `Facturas`, `Renovaciones`, `Bajas`, `Autoconsumo`, `Usuarios`, `Productos`, `Canales` y `Ajustes`.
  - También existen ya las fichas específicas de detalle para cada registro contenido dentro de esas áreas.
  - **REGLA DE ORO**: En adelante, se trata de **mejorar** las secciones anteriores e ir ampliando funcionalidades, pero **NO TENEMOS QUE HACER PÁGINAS QUE YA EXISTEN** desde cero.
- **Infraestructura y Hosting**:
  - Código fuente: **GitHub**
  - Despliegue Frontend/Backend: **Vercel**
  - Base de Datos Relacional: **Vercel Postgres** (No usar Prisma Postgres ni Neon externo directamente, usar siempre la instancia de Vercel Postgres vinculada al proyecto en Vercel).

## 2. Control de Accesos y Roles de Usuario
- **Superadmin Supremo**: El usuario `fjponferrada@sp-energia.com` es el Superadmin y tiene acceso total, absoluto e irrestricto a todas las áreas del software siempre. Es el único que puede ver el menú de "Usuarios", dar de alta nuevos usuarios y asignarles roles, marcas y/o comercializadoras.
- **Responsable de Comercializadora**: Es responsable total de una o varias comercializadoras asignadas. Solo puede operar dentro del contexto de las marcas que cuelgan de las comercializadoras a las que está asociado.
- **Responsable de Marca**: Es responsable total de una o varias marcas. Solo puede visualizar y operar con las marcas a las que esté explícitamente asociado.
- **Backoffice de Marca**: Tiene acceso de gestión y tramitación para la(s) marca(s) a las que esté asociado.
- **Comercial**: 
  - Solo puede ver las secciones: Leads, Contratos, Clientes, Facturas, Renovaciones, Bajas y Productos.
  - **Restricción estricta de edición**: Los comerciales NO tienen permiso para editar NINGÚN registro existente. Solo pueden dar de alta nuevos Leads (tipo Oferta o Contrato).

## 3. Paginación y UI
- **Paginación por Defecto**: En todas las tablas de listados principales del sistema (`leads`, `contratos`, `facturas`, `clientes`, `renovaciones`, `bajas`), el número por defecto a mostrar DEBE ser de **100 elementos por página**.

## 4. Lógica de Leads / Oportunidades (Modal de Alta)
Existen dos flujos muy bien diferenciados en el alta de Leads que los comerciales pueden usar:
- **Flujo "Alta Luz" (Contrato directo)**: 
  - Requiere validación estricta y todos los datos necesarios para generar el contrato y enviarlo a firma.
  - Campos obligatorios: Nombre, NIF, Email, Teléfono, Dirección Completa, Apoderado (si es empresa), Tramitación, Autoconsumo, IBAN, Forma de Pago. Obliga a subir documentación (DNI, Factura, CIE, Escrituras, etc.) y marcar la casilla `VERIFICACION_SOLICITUD`.
- **Flujo "Crear Oferta" (Estudio Comercial)**:
  - Es un flujo ágil para lanzar una oferta sin fricción. Solicita mucha menos información.
  - **NO OBLIGAR** a rellenar dirección, IBAN, firma manuscrita, ni datos de contrato.
  - Se oculta la Verificación final. Solo requiere datos básicos de contacto, suministro (para sacar SIPS), estudio comparativo y la documentación mínima (DNI y Última Factura).

## 5. Ciclo de Vida del Contrato (Estados y Transiciones)
El flujo operativo desde que un Lead se convierte en contrato sigue estrictamente estas fases:
1. **Borrador**: Si un Lead (Alta Luz) se envía a firma (manualmente o automáticamente), se genera el PDF del contrato y se crea un registro en la tabla `Contratos` con estado **Borrador**.
2. **Aceptado**: Cuando el cliente firma el contrato, el estado cambia a **Aceptado**. En este momento, se debe generar un registro en la sección de `Clientes` asignando a este nuevo cliente a la comercializadora correspondiente.
3. **Tramitando**: Cuando se inicia la tramitación del contrato con la distribuidora (mediante fichero switching), el contrato pasa a estado **Tramitando** (esto se podrá hacer manualmente o automáticamente vía scrapping futuro).
4. **Activo**: Cuando la distribuidora confirma la activación, el estado pasa a **Activo** y se actualiza la fecha de inicio.
5. **Finalizado**: Cuando un contrato causa baja, el estado pasa a **Finalizado**.

## 6. Reactivación por Desistimiento (Tipo E1)
- Si un cliente, una vez finalizado su contrato por haberse ido a otra comercializadora, presenta un desistimiento y la distribuidora lo devuelve:
  - Se activa un proceso de reactivación tipo **E1**.
  - Se clona la versión del último contrato que estuvo activo.
  - Se **mantiene la fecha de inicio de permanencia inicial**.
  - Se asigna una **nueva fecha de alta** que coincidirá con la fecha en que la distribuidora comunique la reactivación oficial.

## 7. Espejo de Base de Datos (Airtable a Prisma)
- Todo campo que exista en los backups o definiciones históricas de Airtable (`canales`, `productos`, `clientes`, `modificaciones`, `soporte`, etc.) debe existir en el esquema Prisma y tener su contraparte en la UI si requiere edición o visualización. No simplificar ni eliminar campos sin autorización, ya que Make y automatizaciones externas pueden depender de ellos.

## 8. Lógica Avanzada de Contratos (Histórico)
- **Contratos Multipunto**: Hay clientes (CIFs específicos) que tienen un "contrato multipunto genérico". Para ellos, cualquier CUPS nuevo que dan de alta NO necesita generar PDF ni enviarse a firma. Entran directamente al sistema en estado **Aceptado** y se vinculan a dicho contrato.
- **Histórico de Contratos (Trazabilidad)**: Si se genera un contrato para un CUPS que ya tiene otro contrato activo previo, el nuevo contrato **debe almacenar el ID del contrato inmediatamente anterior** para poder arrastrar y calcular datos.
- **Cambios de Titularidad**:
  - *Subrogación*: Solo se genera un anexo de subrogación (una nueva "versión" del contrato actual) a firmar por el nuevo titular. No se activa hasta confirmación de distribuidora.
  - *Traspaso (Nuevo Contrato)*: Se genera un contrato nuevo desde cero. Cuando se activa, el contrato anterior causa baja.
- **Fecha de Inicio de Permanencia**: 
  - Coincide con la fecha de Alta en la Comercializadora **EXCEPTO** si el trámite es **M1** (Modificación de titular) o **E1** (Desistimiento).
  - Excepción a la excepción: Si se trata de un **M1S** (Modificación administrativa) con "versión vacía" (es decir, proviene de un nuevo contrato y no de una subrogación), la permanencia sí coincide con el alta.
- **Modificaciones Contractuales**: 
  - Las técnicas se gestionan como **M1N** (ej. cambio tarifa o potencia).
  - Las administrativas con subrogación se gestionan como **M1S**.

## 9. Portal de Clientes y Automatizaciones Nativas (No Make/Airtable)
- **Portal de Clientes (Área Privada)**: Los clientes finales de la comercializadora deben poder iniciar sesión en un portal restringido para consultar sus contratos, facturas y datos como cliente. La intención es integrar/reemplazar el uso que actualmente se le da a Softr.

  - Generar PDFs de contratos y enviarlos a firma.
  - Enviar emails de Bienvenida (cuando pasa a Activo).
  - Gestionar las notificaciones de activación de suministros.
  - Generación y envío por email de anexos por modificaciones.

### 9.1 Lógica de Plantillas de Contrato (CORE vs B2B)
- Existen dos modalidades principales de plantillas de contrato: **CORE** y **B2B**.
- **SEPA B2B**: Solo aplicable a empresas y autónomos. En el mandato SEPA de esta plantilla, el cliente renuncia expresamente a su derecho de devolución del recibo (reembolso en 8 semanas).
- **SEPA CORE**: Para particulares (y algunas empresas si no se exige B2B). Mantiene el derecho estándar de devolución de 8 semanas.
- El motor nativo de generación de PDFs del CRM deberá ser capaz de seleccionar la plantilla adecuada en base a la naturaleza del cliente (`Tipo de persona` / `CNAE`) o a la selección explícita del comercial durante el "Alta Luz", inyectando las mismas variables que históricamente se inyectaban vía Make (`{{nombretit}}`, `{{iban}}`, `{{p1e}}`, etc.).

## 11. Regla de Migración y Mapeo de Datos
- **MIGRACIÓN AL 100% (REGLA DE ORO)**: Existe una regla inquebrantable y expresa de que **EL 100% DE LOS CAMPOS** de las tablas de Airtable (en especial `leads`, `contratos`, `facturas`, `instalaciones`, `productos`, `canales`) **tienen que venir cada vez que migremos información** de Airtable a PostgreSQL. 
- **Prohibido perder información**: No estamos aquí para purgar datos ni simplificarlos. La intención es profesionalizar el CRM, pero jamás eliminar información histórica.
- **Uso del campo JSON `airtableData`**: Para cumplir con la regla del 100% sin ensuciar la base de datos relacional, todo el registro bruto de Airtable debe insertarse en el campo `airtableData (JSON)` de las entidades principales. **ESTÁ ESTRICTAMENTE PROHIBIDO** crear campos "basura" o crudos de Airtable directamente en `schema.prisma`. Los campos nativos de Prisma deben mantenerse limpios y los scripts de migración deben mapear los datos relevantes (como NIF, consumos) a estas columnas tipadas. Este JSON sirve como backup absoluto.
- **Mapeo Exhaustivo de Diccionarios**: A pesar de tener el `airtableData`, TODOS los campos documentados en los diccionarios Excel (`diccionario_contratos.xlsx`, etc.) deben ser mapeados obligatoriamente a sus columnas relacionales en Prisma. No se pueden ignorar campos secundarios (como Segundos Apellidos, Teléfonos Extra, Facturas Papel) ni usar valores por defecto (`00000` o `Desconocida`) si el diccionario indica un campo de procedencia.
- **Identificación de Registros Históricos vs Nativos**: No se deben crear campos booleanos estilo `isImported`. Para discriminar en el futuro si un registro es heredado de Airtable o ha sido creado de forma nativa por los comerciales en el nuevo CRM, se debe evaluar la existencia de los campos `airtableId` y `airtableData`. Si son nulos, el registro es 100% nativo.
- **Comisiones de Productos de AED Energía**: En la importación de datos de AED Energía, todos los productos de tarifa `2.0TD` se importan con tipo de comisión "Tramos de Potencia". Los productos que **NO** son `2.0TD` se importan con tipo de comisión "% sobre Margen".
- **Gestión Inteligente de Basura Airtable:**
  - **Archivos y PDFs**: Cualquier campo de tipo archivo (CIE, DNI, Contrato Firmado, etc.) no se debe almacenar como simples URLs dispersas en el contrato. Se deben crear registros en la tabla global `Document` vinculados al registro padre.
  - **Mails de Acceso**: Los campos como `Email Comercial`, `Email Gerente`, `Email Sup Canal` usados históricamente en Airtable para dar permisos de vista a usuarios externos NO se importan. La seguridad en el nuevo CRM se gestiona por roles, marcas y comercializadoras nativas.
  - **Asignación de Comercial/Canal**: El identificador real de Comercial o Canal que venga de Airtable SÍ debe usarse para poblar las claves foráneas (`userId`, `channelId`) en los Contratos y Leads, de modo que hereden los permisos nativamente.
  - **Basura literal**: Se pueden ignorar de forma excepcional campos basura evidentes como `P1PM copy copy`, o fórmulas temporales para Make, siempre y cuando NO supongan información de negocio real.

## 14. Lecciones Aprendidas y Prevención de Errores (Edge Cases)
Para evitar tropezar dos veces con la misma piedra durante el desarrollo continuo del CRM, el desarrollador (IA) debe tener siempre presente los siguientes casos especiales resueltos:

1. **Relación Contract-SupplyPoint**: En Prisma, el modelo `Contract` tiene un único punto de suministro vinculado (`supplyPoint`, **en singular**), mientras que `Client` tiene múltiples (`supplyPoints`, **en plural**). Es crítico no usar `supplyPoints` en los `include` de Prisma cuando se consulta un contrato, ya que provocará un `PrismaClientValidationError`.
2. **Prioridad de Nombres y Trampas de Airtable**: Al importar clientes, es vital crear una función limpiadora (`trim()`) que descarte celdas que solo contengan espacios en blanco (ej. `" "`), ya que Airtable permite su inserción por error humano. La regla B2B/Físico es: Si `"Primer Apellido"` existe, se usa `"NOMBRE Y APELLIDOS"`. Si no, se asume Empresa y se usa `"NOMBRE/RAZON SOCIAL"`, con un fallback a `"NOMBRE Y APELLIDOS"` o `"Nombre completo Titular"` si la razón social falla.
3. **Manejo Defensivo de Arrays en Airtable**: Campos relacionales o de búsqueda (Lookups) como `"Comercial"` o `"CANAL"` viajan a través de la API de Airtable encapsulados en Arrays (ej: `["rec3fxKOf4kKpcbds"]` o `["Pablo Remacha"]`). **NUNCA** se debe asumir que son textos directos. Siempre hay que extraer la primera posición (`[0]`) y forzar un `String(...)` antes de renderizar en UI o aplicar métodos de texto como `.slice()` o `.toUpperCase()`, evitando así cuelgues del Front-End (`TypeError`).
4. **Mapeo Relacional de Comerciales (Fin del Diccionario Hardcodeado)**: Inicialmente se intentó usar un diccionario estático en código para traducir iniciales (`JP -> Javier Ponferrada`). Esto es una **mala práctica**. La regla oficial es que los comerciales se importan desde la tabla original `USUARIOS` de Airtable directamente a la tabla `User` de Prisma (filtrando solo los que tienen `CANALES LINK`, es decir, los verdaderos comerciales). Para saber qué comercial vendió un contrato/lead, se busca en Prisma el usuario cuyo `email` coincida con el array `airtableData['Email Comercial'][0]`, y se vincula relacionalmente (`lead.userId = user.id`). Así, la UI simplemente renderiza `lead.user.name`.
5. **CUPS MOCK y Validaciones SIPS (Ingebau)**: Durante las importaciones, si un contrato/lead carece de CUPS temporalmente, los scripts inyectan un identificador `CUPS_MOCK_...` para no romper relaciones nulas en Prisma. Sin embargo, al consultar a SIPS (Ingebau), el motor de back-end debe detectar estos MOCKS, ignorarlos y tratar de escarbar en `airtableData['CUPS2']` antes de lanzar la petición. Ingebau colapsará si recibe un CUPS MOCK.
6. **Interpretación de Potencias (kW vs W)**: En Airtable existen los campos `P1C`, `P2C`, etc. que contienen la potencia en kW (con separadores de coma o punto). Existen campos paralelos llamados `P1C W` que contienen fórmulas que lo multiplican por 1000 para obtener Vatios. En nuestro software **debemos ignorar por completo las fórmulas en vatios (P1C W)** y utilizar SIEMPRE los campos originales `P1C`, parseando las comas por puntos, asumiendo su valor original en kW para evitar alteraciones o anomalías.
7. **Caché Persistente de Prisma en Next.js (Error SELECT *)**: Tras alterar el `schema.prisma` y eliminar columnas obsoletas (ej. `customFee`, `autoconsumo`), ejecutar `npx prisma generate` **NO es suficiente**. Next.js mantendrá en memoria caché la instancia antigua del cliente de Prisma. Al ejecutarse operaciones como `prisma.lead.findMany()` en cualquier página, Prisma intentará hacer un `SELECT *` solicitando la columna que ya no existe y lanzará errores `PrismaClientKnownRequestError` de tipo "The column X does not exist in the current database". Para solucionarlo en **todas** las páginas de golpe, es de obligado cumplimiento **detener la terminal (Ctrl+C) y arrancar en frío el proceso del servidor (`npm run dev`)** tras alteraciones destructivas en el esquema.
8. **Semántica de la Tabla USUARIOS de Airtable**: Al tratar con la tabla de usuarios heredada, se ha aprendido que: `Email Link` o `Email` coinciden con el email de acceso a Softr y será el login del CRM. `Nombre2` es el nombre real y completo del usuario comercial. `Código` es su identificador de comercial (ej. `PR`). `Supervisor canal` marca a los responsables de canal. El resto de emails y comisiones quedan relegados porque el CRM gestionará la seguridad por Roles y Marcas/Canales nativos.
9. **Semántica de la Tabla CANALES de Airtable**: De la tabla de canales heredada se extrae que la relación más crítica a mantener es la de `PRODUCTOS` (determina qué tarifas/productos puede visualizar y vender cada canal) y la de `COMERCIALES` (qué usuarios pertenecen a este canal). Además de su `Nombre` y `Código Canal`, existen emails para notificaciones (`Email Supervisor`, `Email Administracion`) que se mantendrán para envíos automáticos. Los campos de comisiones o automatismos genéricos (`% Comisión`, `GEN_CONTRATO_AUTO`) no se importan de forma dura, pues el CRM lo gestionará con su propio motor.
10. **Idempotencia y Purga en Migraciones**: Para evitar duplicidades de productos, clientes o facturas fantasmas, el script de migración SIEMPRE debe realizar un borrado en cascada (`deleteMany` o limpieza de base de datos) antes de insertar los nuevos datos, ya que no se pueden garantizar identificadores únicos estables que permitan hacer `upsert` seguro de forma masiva sin un UUID.
11. **Tipos de Comisión (POWER_TIERS vs MARGIN_PERCENTAGE)**: La forma en que se comisiona un contrato depende de la `Tarifa` del producto subyacente. Si la tarifa es `2.0TD` (Residencial/PYME), el tipo de comisión SIEMPRE debe fijarse internamente a `POWER_TIERS` (ramos por tramo de consumo, ej: <3000 kWh = 30€). Para cualquier otra tarifa (`3.0TD`, `6.1TD`, etc.), el tipo es `MARGIN_PERCENTAGE` (se paga un % sobre el Fee o margen industrial). Esta es una regla absoluta en el CRM que no debe depender de banderas manuales en Airtable.
12. **Pureza de Base de Datos y Prohibición de Joins Dinámicos para Precios Históricos**: Durante la migración, la base de datos PostgreSQL debe consolidar la "foto fija" del contrato (sus precios firmados, fees, excesos, etc.) directamente en los campos nativos de la tabla `Contract` y `Lead` (ej: `p1e`, `p1p`, `fee`, `pexc`). **Queda ESTRICTAMENTE PROHIBIDO heredar, calcular o leer precios dinámicamente haciendo join con el Catálogo de Productos (`Product`) en la UI o en los procesos de facturación.** La tabla de Productos es dinámica y sus precios cambian; los contratos históricos NO deben alterarse si cambia el catálogo. Cualquier referencia en el código a `product.p1e` o `customFee` en lugar de `contract.p1e` o `contract.fee` romperá la base de datos porque esas columnas "custom" obsoletas han sido purgadas del esquema de Prisma.
13. **Prioridad de Snapshots Estáticos frente a Lookups Dinámicos (Airtable)**: En Airtable, existen columnas que son lookups dinámicos (`P1E (from PRODUCTOS)`, `P1P (from PRODUCTOS)`) and columnas que son snapshots puros del momento de la firma (`P1P`, `P2P`, etc.). En la migración, se ha descubierto que Airtable SÍ guarda un snapshot estático independiente para los precios de **Potencia** (`P1P`), pero NO lo hace para la **Energía** (`P1E`). Por tanto, el script de migración SIEMPRE debe priorizar el snapshot puro para la potencia (`f['P1P'] || f['P1P (from PRODUCTOS)']`), para asegurar que prevalece el precio histórico (ej: 40.39€) firmado por el cliente en el contrato original, evitando que sea sobrescrito por el precio lookup vinculado al producto actual del catálogo.
14. **Regla 14 (Vinculación de Facturas por Contrato Airtable)**: Las facturas en Airtable están vinculadas al contrato a través de una columna Lookup Múltiple en `Contratos` (llamada `FACTURASfldaa54Nua6LxjjX7`). La importación de facturas DEBE recorrer esta matriz de IDs y hacer *fetch* individual por ID para evitar los límites de paginación (`maxRecords`) y asegurar que se descarga el "árbol completo" de facturas de un contrato.
15. **Regla 15 (Capitalización en Prisma)**: Cualquier inclusión relacional en el código Prisma (`include: {}`) debe hacerse estrictamente en *camelCase* (`contract`, `supplyPoint`, `invoice`, `product`, `lead`), de acuerdo a la configuración del esquema. El uso de PascalCase en los includes provocará `PrismaClientValidationError`.
16. **Regla 16 (Datos del Representante/Apoderado en Airtable)**: Según el diccionario de datos de Airtable, para los contratos B2B (Personas Jurídicas), los campos `Nombre Contacto`, `Apellidos Contacto` y `NIF Contacto` almacenan en realidad los datos del Representante Legal / Apoderado. El script de migración mapea estos campos directamente a `representativeName`, `representativeVat`, etc., para que no aparezcan vacíos.
17. **Regla 17 (Nomenclatura y Mapeo de Conceptos SVA)**: El campo `Servicio` en Airtable devuelve un ID de registro interno (`rec...`). Este ID debe ser mapeado explícitamente a su nombre real (ej. "Alta Nueva (25 €/alta)") mediante un diccionario estático en la migración y el frontend, en lugar de intentar extraerlo del campo fórmula `Producto y Servicio` con expresiones regulares, ya que este último a menudo no incluye comillas.
18. **Regla 18 (Tipos de Producto)**: En Airtable, el `Tipo de producto` es una cadena de texto explícita (ej. "Precio indexado Pass-Through", "Precio fijo único"). En lugar de reducirlos a constantes binarias (`FIX` o `INDEXED`), Prisma y la UI almacenan y renderizan la cadena literal exacta, garantizando mucha más claridad a nivel comercial.
19. **Regla 19 (Paginación Oculta de Airtable)**: La API de Airtable devuelve un máximo estricto de 100 registros por consulta. Queda TERMINANTEMENTE PROHIBIDO realizar descargas directas sin paginación para tablas maestras (`PRODUCTOS`, `CANALES`, `USUARIOS`, etc.). El script de migración siempre debe implementar un bucle `do-while` que inyecte el parámetro `offset` en la URL hasta que ya no haya más páginas, evitando así truncamientos catastróficos de catálogos (ej. perder las tarifas 2.0TD por estar a partir de la fila 100).
20. **Regla 20 (Limpieza de Espacios y Productos Fantasma)**: Los usuarios de Airtable introducían espacios finales por error (ej. `"AED 24h "`). Para evitar que el CRM falle al buscar el producto al importar un contrato, la regla de oro es aplicar siempre `.trim()` a los nombres de productos antes de cualquier `findFirst`. Además, si un producto referenciado en un contrato realmente no existe, el CRM debe crear un "Producto Fantasma" para evitar orfandad, al cual **se le debe asignar obligatoriamente la Tarifa** que venga en la fila de dicho contrato (`f['Tarifa']`) y su tipo de comisión correspondiente, en lugar de dejar la tarifa a `null`.
21. **Regla 21 (Campos Obsoletos del UI y Nuevo Motor de Comisiones)**: Las columnas históricas de `IP` (Indexado Potencia), `FC` (Fijo Comercial) y `Costes Estimados (P1E_Coste...)` han sido purgadas del `schema.prisma` debido al "Nuevo motor de comisiones". El Frontend NO DEBE renderizar "campos fantasma" en los formularios (ej. `ProductModal.tsx`) que intenten enviar estos datos al Backend, ya que causará un Error 500 en Prisma por columna inexistente. Solo se renderizarán y editarán campos reales de BD.
22. **Regla 22 (Fechas de Facturación - Legacy vs DateTime)**: El esquema de Prisma para `Invoice` contiene campos de fecha fuertes (`billingStart`, `billingEnd`) y campos de cadena legacy de Airtable (`desde`, `hasta`, `desdeEA`, `hastaEA`). Para que la visualización UI (ej. "Periodo Energía") legacy no se rompa o muestre "-", el script de migración debe rellenar ambos tipos de campos simultáneamente.
23. **Regla 23 (SuperAdmin Global)**: El usuario super administrador por defecto del CRM es `fjponferrada@sp-energia.com`. Al realizar migraciones, los contratos y leads no asignados explícitamente se volcarán bajo este usuario en lugar de cuentas ficticias (como `admin@migracion.com`), garantizando así que los datos no se queden "huérfanos".
24. **Regla 24 (Migración y Asignación de Canales)**: Airtable almacena los canales de distribución en una tabla independiente (`CANALES`). La importación de datos descarga estos canales y los inserta en el modelo `Channel` (usando el Record ID de Airtable como clave primaria `id` y `code`). Al procesar un contrato, si se detecta que pertenece a un canal y tiene un comercial asociado (`Email Comercial`), el CRM vinculará automáticamente el usuario a su respectivo `channelId`, permitiendo que la UI refleje el canal del agente comercial con claridad.
25. **Regla 25 (Catálogo de Productos por Canal)**: Los Canales tienen una relación Muchos a Muchos (`Many-to-Many`) con los Productos. Un canal solo tiene permiso para vender los productos que se le hayan habilitado en su matriz `PRODUCTOS`. En la migración, se debe cruzar el array de IDs original con los Productos ya importados, y en la UI, el Modal de Canal debe listar explícitamente el catálogo autorizado para dicho canal.
26. **Regla 26 (Limpieza e Importación de Usuarios/Comerciales)**: En la migración de datos, se purga la tabla de `User` (a excepción estricta del Superadmin `fjponferrada@sp-energia.com` o cuentas críticas de sistema) para luego importar masivamente desde la tabla `USUARIOS` de Airtable. **Filtro activo**: solo se traen aquellos registros que disponen de `CANALES LINK` (los verdaderos comerciales en activo). **Asignación de Roles automática**: si el usuario tiene marcado el checkbox `Supervisor canal` (`true`) en Airtable, se le da el rol de `CANAL` (`isChannelSupervisor = true`). De lo contrario, se le asume agente raso y recibe el rol de `COMERCIAL`.
27. **Regla 27 (Seguridad Estricta por Marcas Asignadas)**: Aunque el esquema de Prisma obliga a que cada usuario tenga una `brandId` como marca primaria o por defecto, **el acceso real al sistema está dictado EXCLUSIVAMENTE por sus Marcas Asignadas (`assignedBrands`)**. Si a un usuario (CANAL o COMERCIAL) se le desmarcan todas las marcas autorizadas en su ficha, el sistema arrojará un error 403 / "Acceso denegado" en el Login, ignorando la marca primaria subyacente. Para mantener la coherencia en usuarios migrados antiguamente (que solo tenían `brandId`), un script inicial de "backfill" copió su marca primaria dentro de su array de `assignedBrands`.
28. **Regla 28 (Jerarquía de Permisos Visuales: Companies -> Brands)**: En la interfaz (UI) y a nivel lógico, el acceso a una marca (`assignedBrands`) es contingente a tener acceso a su comercializadora padre (`companies`). Si a un usuario no se le asigna la entidad matriz en su relación `companies`, la UI no renderizará las marcas dependientes de la misma, garantizando que los permisos se otorgan siempre de forma jerárquica de arriba abajo.
29. **Regla 29 (Campos no estructurados - `contractData` en Leads)**: El esquema relacional Prisma para el modelo `Lead` debe mantenerse limpio de campos accesorios, temporales o inestables (ej. `titularStreetType`, `contactRole`, etc. que vienen de formularios dinámicos). Todo campo que no pertenezca estrictamente al modelo CORE relacional del Lead debe anidarse de forma segura dentro del campo JSON `contractData`.
30. **Regla 30 (Validación de Tipos Prisma en Next.js)**: Cualquier alteración estructural al `schema.prisma` (renombrar relaciones, eliminar columnas) requiere no solo `npx prisma generate`, sino que es de obligado cumplimiento matar los procesos y limpiar la caché de compilación de Next.js. De lo contrario, Turbopack utilizará la declaración de tipos obsoleta en caché y el build fracasará con un `PrismaClientValidationError`.
31. **Regla 31 (Pureza de API en Endpoints)**: Los endpoints de la API (ej. `/api/products`) deben estar estrictamente saneados. Propiedades antiguas o variables de configuración legacy que ya no existan en el esquema oficial de Prisma (como `ip`, `fc`, `gasIncluido`) NO pueden ser inyectadas en las cláusulas `data: {}` de creación/actualización. Hacerlo provocará un error catastrófico de tipado *"Object literal may only specify known properties"* durante la compilación.
32. **Regla 32 (Aislamiento Estricto Multi-Marca)**: Para garantizar el funcionamiento como SaaS Multitenant y el cumplimiento de RGPD, los identificadores como `vatNumber` (DNI), `cups` o `invoiceNumber` **NO SON GLOBALES**. Su unicidad está delegada y combinada estrictamente con el `brandId`. Un mismo DNI o CUPS puede existir múltiples veces en la base de datos siempre que pertenezcan a marcas diferentes. Todo el backend (vía `findFirst`), listados UI (vía `VisibilityFilters`) y operaciones CRUD deben inyectar ineludiblemente el `brandId` del usuario o sesión actual para garantizar que jamás haya fuga o cruce de datos entre comercializadoras distintas.
33. **Regla 33 (Única Instancia de Prisma en Next.js/Vercel)**: En entornos Serverless como Vercel, está terminantemente prohibido instanciar `new PrismaClient()` manualmente dentro de los Server Components (`page.tsx`) o endpoints de la API. Hacerlo agotará el *connection pool* de la base de datos y causará errores `PrismaClientKnownRequestError` o fallos de lectura de la variable `DATABASE_URL` frente a `PRISMA_DATABASE_URL` / `POSTGRES_URL`. Todos los archivos deben inyectar ineludiblemente la instancia global y cacheada de Prisma importándola exclusivamente desde el archivo central: `import { prisma } from '@/lib/prisma'`.
34. **Regla 34 (Contexto de Migración Airtable -> AED)**: Todo el contenido y registro histórico proveniente de los backups de Airtable pertenece 100% en exclusiva a la comercializadora semilla "AED Energía" y a su marca homónima. Al ejecutar scripts de migración, **ESTÁ PROHIBIDO** purgar o borrar las tablas de `Company` o `Brand`. El script de importación debe buscar la marca "AED" existente en la base de datos y vincular relacionalmente el 100% de los Usuarios, Clientes, Contratos y Leads importados a dicha marca.
35. **Regla 35 (Gestión Autónoma del Servidor de Desarrollo)**: Cada vez que el servidor local de desarrollo (`npm run dev`) se caiga, el ordenador se reinicie, o el usuario pida explícitamente "arrancar/reiniciar el servidor", el agente de IA **debe** encargarse directamente de lanzar el comando de ejecución en segundo plano sin delegarlo en el usuario.
36. **Regla 36 (Fusión de Oportunidades y Leads Airtable en CRM SP)**: 
    - **Flujo original en AED (Airtable)**: Se usaban dos tablas distintas. `OPORTUNIDADES` (datos básicos para lanzar ofertas) y `LEADS` (si aceptaba la oferta, se pedían datos completos para generar borrador de contrato PDF). Si lo firmaba, pasaba a `CONTRATOS`. Luego las facturas se ataban al contrato. Relación estricta: Lead -> Contrato -> Facturas.
    - **Flujo nuevo en CRM SP Energía**: Ambas fases (oportunidad rápida y borrador de contrato completo) se fusionan y unifican bajo la tabla y sección única de `Leads`. Desde esta única página, el usuario puede lanzar una oferta básica ("Crear Oferta") o tramitar el contrato completo ("Alta Luz") como indica la Regla 4. Al migrar los datos de Airtable, se debe tener en cuenta esta fusión unificadora.
37. **Regla 37 (Aislamiento Multimarca de Canales)**: Históricamente concebidos como entidades globales, los Canales (`model Channel`) están estrictamente aislados y pertenecen a una única marca (`brandId`). El backend, las vistas (UI) y las consultas Prisma deben filtrar rigurosamente por el `brandId` del usuario activo (`getChannelVisibilityFilter()`) para evitar que responsables de distintas comercializadoras vean canales de su competencia. Las migraciones y creaciones de nuevos canales inyectarán siempre el `brandId` adecuado.
38. **Regla 38 (Estrategia de Migración y Sincronización Incremental)**: El proceso de volcado masivo de datos desde Airtable se divide en dos enfoques para garantizar seguridad, integridad y tolerancia a fallos:
    - **Migración Masiva (`migrate-airtable.ts`)**: Proceso destructivo de "borrón y cuenta nueva". Elimina toda la base de datos (excepto administradores clave) e importa el 100% de la jerarquía (Compañías, Marcas, Canales, Productos, Usuarios, Contratos, Leads y Facturas). Las facturas se precargan en memoria (un solo fetch global) para evitar el colapso de la API por peticiones N+1. Cualquier error individual en un contrato se atrapa de forma silenciosa y se vuelca en un log persistente (`migration_errors.json`).
    - **Sincronización Incremental (`sync-missing.ts`)**: Proceso no destructivo de "recuperación". Diseñado para reparar o importar contratos faltantes tras resolver errores del primer volcado. Compara los `airtableId` de la BD local con los de Airtable, detecta el delta (los ausentes) y procesa única y exclusivamente esos registros sin borrar nada del entorno productivo, generando su propio log `sync_errors.json` en caso de fallos. Todas las búsquedas previas de CUPS, DNI (`vatNumber`) o número de factura deben incluir ineludiblemente el contexto de la marca (`brandId` o `clientId`) como dicta la Regla 32.
39. **Regla 39 (Prioridad del CIF Original y Mapeo Completo del Cliente)**: El verdadero identificador fiscal en los contratos históricos se extrae del campo de texto `CIF`, no de columnas derivadas o links como `Copia de CIF link`. Al migrar contratos, si el cliente (Client) no existe en BD por coincidencia estricta de `vatNumber` y `brandId`, se debe crear la nueva ficha absorbiendo el 100% de la información comercial proveniente del registro de Airtable del contrato (incluyendo `DOMICILIO SOC`, `CP SOC`, `POBLACION SOC`, `IBAN`, `CNAE`, etc.). Jamás se deben dejar fichas de clientes huérfanas de información por crear un cliente de emergencia sin DNI.
40. **Regla 40 (Preservación Detallada de Estados de Rechazo)**: Los estados de "Rechazado" en Airtable incluyen casuísticas que deben ser rastreables. El importador NO debe colapsar todos los estados de rechazo en un macroestado global `RECHAZADO`. Se deben generar y mantener los subestados específicos (`RECHAZO_DISTRIBUIDORA`, `RECHAZO_COMERCIALIZADORA`, `RECHAZO_RIESGOS`, `RECHAZADO_POR_CLIENTE`) en el campo `status`, dotando a la UI y a los filtros de granularidad total para la toma de decisiones.
41. **Regla 41 (Mapeo Estricto de Observaciones Contractuales)**: Toda anotación en el campo `Observaciones` (u `OBSERVACIONES`) de Airtable para un contrato debe volcarse incondicionalmente en la propiedad `internalComments` del modelo `Contract` en Prisma. Esta información es crítica y el Front-End está programado para renderizarla a modo de tooltip interactivo (`MessageSquare`) sobre el propio listado de contratos, sin necesidad de entrar a la ficha detalle.
42. **Regla 42 (Paginación SSR y Optimización de Consultas)**: Debido a que la base de datos supera holgadamente los miles de registros, queda terminantemente prohibido volcar o renderizar todas las filas de golpe (`findMany()` sin límites) hacia el cliente de React (UI). Todas las tablas maestras (Clientes, Contratos, Facturas) deben emplear Server Actions nativas con delegación de paginación a nivel de SQL (`skip` / `take`). Las estadísticas generales superiores (ej: Total Activos, MWh total) deben resolverse mediante consultas agregadas independientes para desacoplar el rendimiento de la tabla visual de la carga de cálculo matemático.
43. **Regla 43 (Cálculo Dinámico de Márgenes en Facturas)**: Para el dashboard y analítica económica, se ha integrado la fórmula de cálculo de márgenes reales (Margen Ingebau vs Margen Estimado). Es fundamental que el campo `margin` se calcule y consolide nativamente en la base de datos para cada factura importada, saneando valores anómalos.
44. **Regla 44 (Prevalencia del Fee de Contrato sobre el Producto)**: Al calcular márgenes o comisiones de una factura, el sistema SIEMPRE debe priorizar el `fee` extraído de la tabla `Contract` asociada a esa factura. El `fee` genérico del catálogo de productos (`Product`) solo se usará si el contrato no tuviese uno definido. Esto previene que alteraciones modernas en el precio base del catálogo corrompan la rentabilidad real que se le firmó al cliente en su día.
45. **Regla 45 (Cifra de Negocio y Abonos en el Análisis Económico)**: Para representar fielmente la "Facturación" o volumen de ventas en el Dashboard, JAMÁS se usará el campo `totalAmount` (que incluye impuestos). La facturación oficial es siempre la suma de la base imponible (`baseImponibleIva`). Además, el sistema invierte automáticamente el signo (haciéndolo negativo) de las bases imponibles si el `Tipo Factura` detectado es un `Abono`, asegurando la precisión contable de la cifra de negocio mensual.
46. **Regla 46 (Unificación del Panel de Control)**: Toda la analítica de negocio (Evolución de Cartera MWh, Altas, Bajas, Facturación vs Margen y Rendimiento) se debe condensar siempre en la pantalla principal del `Dashboard`. La política de interfaz exige evitar la fragmentación en sub-páginas analíticas para ofrecer al perfil directivo una visión global inmediata en portada.
47. **Regla 47 (Lógica de Falsas Bajas y Periodo de Gracia)**: A efectos de cálculo de KPI en el Dashboard y renderizado en la tabla principal de Contratos, existe la figura de la "Falsa Baja". Si un contrato finaliza (`status = 'BAJA'`), pero para ese mismo Punto de Suministro (CUPS) se activa otro contrato posterior en un plazo inferior a 30 días (Periodo de Gracia), se considera una continuidad comercial (ej. renovación o modificación) y NO una fuga de cliente. Estas "Falsas Bajas" se excluyen matemáticamente del recuento de churn del dashboard y se ocultan visualmente de los listados para no contaminar la operativa del comercial.
48. **Regla 48 (Histórico Estricto de Potencias Contratadas)**: En la misma línea que la Regla 12 sobre la preservación de precios de energía estáticos, las potencias contratadas en el momento de la firma (P1 a P6 en kW) deben almacenarse como un "snapshot" estático dentro del propio modelo `Contract` (`p1c`, `p2c`... `p6c`). Aunque un Punto de Suministro subyacente cambie su potencia actual con el tiempo (por una subida de potencia, por ejemplo), el registro histórico del contrato debe mostrar inalterable la potencia exacta con la que fue generado o importado originariamente.
49. **Regla 49 (Compatibilidad de Prisma Accelerate y Adaptador PG)**: En entornos Serverless de Vercel, si se utiliza el adaptador nativo `@prisma/adapter-pg` en combinación con el módulo de Node `pg` (para hacer pooling interno), **queda estrictamente prohibido** utilizar enlaces de Prisma Accelerate (`prisma+postgres://...`). La librería `pg` no soporta proxies Edge por el puerto 5432 y genera un `ETIMEDOUT` masivo. En esta configuración, Vercel debe conectarse SIEMPRE mediante el enlace directo (`postgres://...`). Para poder usar Accelerate, es requisito indispensable eliminar el adaptador `pg` y migrar a la extensión oficial `@prisma/extension-accelerate`.
50. **Regla 50 (Latencia Geográfica Vercel vs Prisma)**: Las funciones Serverless de Vercel se despliegan por defecto en Washington D.C. (`iad1`). Si la base de datos de Prisma Postgres está en Europa (ej. París `eu-west-3` o Frankfurt `eu-central-1`), el viaje transatlántico de los datos añadirá 1-2 segundos de "lag" en cada navegación de la interfaz. Para lograr tiempos de carga instantáneos (5-10 ms), **es de obligado cumplimiento** asegurar que la `Function Region` en el panel de Vercel coincida geográficamente con la región del Cloud Provider de Prisma (ej. Vercel `cdg1` para París o `fra1` para Frankfurt).
51. **Regla 51 (Límites de Facturación y Bloqueos Prisma Postgres)**: El plan gratuito (Free) de un Workspace en Prisma Cloud tiene un límite estricto de 100.000 consultas/operaciones al mes. Importaciones masivas de datos (como volcados SQL de Airtable de +1GB) sobrepasarán este límite en cuestión de minutos, provocando que la plataforma bloquee el acceso a la base de datos y la UI lance errores persistentes del tipo `Invalid 'prisma.entity.findMany()' invocation`. Toda migración histórica debe ejecutarse en bases de datos que cuenten con un plan de pago activo (ej. Starter 10$/mes) aplicado correctamente sobre el Workspace destino.
52. **Regla 52 (Almacenamiento en la Nube de Archivos y PDFs - Cloudflare R2)**: Para el alojamiento físico de documentos pesados (PDFs de facturas, contratos firmados, DNI, CIEs, etc.), el sistema utiliza como infraestructura base **Cloudflare R2** (un clon de Amazon S3 sin costes de salida de red). Queda estrictamente prohibido guardar archivos codificados en Base64 en la base de datos relacional de Prisma Postgres. El flujo de subida de cualquier archivo debe enviarse al endpoint `/api/upload`, el cual lo inyectará en Cloudflare R2 y devolverá una URL pública (`https://pub-fca...`). Esta URL es lo único que se almacena en PostgreSQL (generalmente mediante el modelo global `Document` o campos nativos como `pdfUrl`).
53. **Regla 53 (Procesamiento Nativo de Ficheros SCTD y Prevención de Sobreescritura)**: La ingesta de archivos XML de distribuidora (procesos C1, M1, A3, etc.) se hace de forma 100% nativa en el CRM para eliminar dependencias externas (Make/Drive). El archivo se procesa en memoria y se envía directamente a Cloudflare R2 para auditoría. Además, el parser incluye un sistema estricto de Warnings: Si un XML trae una fecha de activación para un CUPS, pero el contrato en la base de datos ya tiene ese campo de fecha lleno, el sistema **no sobreescribirá** la fecha para evitar daños por duplicidad, y en su lugar generará un registro de "Warning" para que el Backoffice lo revise manualmente en la página de Switching Warnings.
54. **Regla 54 (Fuente de Verdad para Estructuras XML CNMC)**: El parser de archivos XML no se diseña en base a suposiciones ni scripts antiguos, sino que debe programarse basándose estrictamente en los esquemas XSD oficiales publicados por la CNMC (Sistema de Comunicación de Datos o SCTD). Toda la documentación técnica, PDFs de los formatos (A3, C1, B1, etc.) y los 91 archivos `.xsd` oficiales se encuentran alojados y deben ser consultados en la ruta de red local: `Z:\AED\Switching\CNMC - E - V3.0 2024.05.16`.
55. **Regla 55 (Consistencia de Estados del Ciclo de Vida y Tipado Prisma)**: El estado final de un contrato cuando causa baja es estrictamente `Finalizado`, NUNCA `BAJA`. Las actualizaciones de estado en el backend (ej. al ingerir archivos XML de distribuidora) deben usar `Finalizado`. Además, las alteraciones o inyecciones de fecha deben apuntar siempre a las variables reales del esquema de Prisma (ej. `terminationDate` en lugar del antiguo `endDate`), ya que Prisma interceptará cualquier propiedad desconocida y abortará la transacción entera devolviendo un error 500, dando la falsa impresión de que un proceso (como la ingesta de archivos) ha funcionado a medias si el error es capturado silenciosamente.
56. **Regla 56 (Arquitectura Desacoplada para Facturas F1)**: Los archivos de facturación de peajes (proceso F1 de la CNMC) NO deben estar ligados estáticamente a un `contractId` en el momento de su ingesta. Deben asociarse únicamente al `supplyPointId` (CUPS). La relación entre un F1 y el contrato o comercializadora responsable se calculará **dinámicamente en tiempo de lectura** (tanto en la vista general como en la ficha del contrato). Este cálculo se realiza cruzando el periodo de facturación (`fechaInicio` y `fechaFin` del XML F1) con el periodo de vigencia del contrato (`activationDate` a `terminationDate`), previniendo que modificaciones retroactivas de contratos dejen facturas F1 secuestradas en contratos inactivos.
57. **Regla 57 (Protección de Ingesta y Exclusión de Procesos Cruzados)**: Para evitar colapsos en el parseo y contaminación de datos en la base de datos de eventos de Switching, el motor de ingesta de "Switching Warnings" bloqueará estrictamente cualquier XML cuyo `<CodigoDeProceso>` pertenezca a `F1` (y viceversa, el módulo de F1 bloqueará procesos que no sean F1). Además de basarse en la raíz del XML (ej. `<MensajeFacturacion>`), el chequeo en profundidad de las etiquetas de cabecera previene vulnerabilidades de inserción accidental por parte del usuario final.
58. **Regla 58 (Búsqueda de CUPS en Contratos Históricos Airtable)**: Los contratos y leads heredados de Airtable se volcaron enteros en el campo `airtableData`, por lo que el CUPS no siempre está en el modelo relacional nativo (`SupplyPoint`) si no se migró de forma perfecta en su día. Para evitar errores de "CUPS no encontrado" en procesos automáticos (ej. importar XML de Switching), el motor de búsqueda SIEMPRE debe realizar un fallback buscando el `codigoSolicitud` u ojeando el JSON `airtableData` en la tabla `Lead` si la búsqueda directa por la tabla `SupplyPoint` falla. Además, el CUPS de los ficheros de la CNMC puede llegar con 22 caracteres (incluyendo sufijos como `0F` o `1F`), por lo que las búsquedas en base de datos deben limitarse siempre a los primeros 20 caracteres del CUPS (`cups.substring(0, 20)`).
59. **Regla 59 (Procesos de Baja y Reposición CNMC - Diferenciación B1, B2 y E2)**: Existen tres flujos que causan confusión y que deben ser tratados en el importador de forma diferenciada para evitar cierres prematuros de contratos o errores de estado `CONTRATO_NO_TRAMITANDO`:
    - **B1 (Baja por Impago)**: En su paso `02` (Aceptación), debe marcar nativamente el contrato con `suspendido = true`. En su paso `05` (Activación Baja), cambia el estado del contrato a `Finalizado`.
    - **B2 (Baja a iniciativa del distribuidor)**: Según el estándar CNMC, el B2 consta de un único mensaje (paso `05`). Cuando llega, debe forzar el contrato a estado `Finalizado`. (Nota histórica: Internamente el equipo comercial a veces llama "B2" a las reposiciones, pero a nivel código se debe tratar estrictamente como baja).
    - **E2 (Reposición de Suministro)**: Es el verdadero proceso de reconexión (archivos como `ActivacionReposicion.xsd`). Cuando se procesan archivos E2 (pasos 14, 15, etc.), el sistema simplemente debe enlazarlos al contrato en estado `ACTIVO` sin alterar su estado ni la marca de `suspendido`.
60. **Regla 60 (Consistencia en la Edición de Contratos)**: El modal de edición rápida de contratos (`EditContractModal`) nunca debe intentar extraer variables directamente de un registro básico o vacío si este proviene de una visualización simple (row de tabla). Debe hidratarse previamente usando la misma información detallada (includes) que emplea la vista de Consulta, garantizando que el usuario jamás se encuentre con un formulario vacío que pueda sobreescribir datos accidentalmente con valores nulos o "borrar" los campos del contrato.
61. **Regla 61 (Generación Segura de Contratos Word a PDF vía DocuSign)**: Para facilitar la labor administrativa, las plantillas base de los contratos se mantienen en formato Word (`.docx`) con etiquetas de variables dobles (`{{campo}}`). Sin embargo, por seguridad jurídica, los comerciales JAMÁS deben descargar o interactuar con el formato `.docx` editable. El CRM sustituye las variables internamente y de forma invisible envía el Word resultante a DocuSign como un "Sobre Borrador" (`created`). DocuSign hace las veces de conversor transformándolo en un PDF inalterable de alta fidelidad. El CRM descarga este PDF para almacenarlo en R2 y descarta el borrador. Esto garantiza máxima seguridad legal sin depender de convertidores frágiles locales o LibreOffice.
62. **Regla 62 (Compatibilidad de Next.js Turbopack con librerías AMD/UMD)**: El motor de empaquetado ultra rápido de Next.js (Turbopack) tiene problemas de compilación estática (`TP1200 unsupported AMD define() dependency element form`) cuando intenta procesar paquetes antiguos que utilizan módulos AMD/UMD (por ejemplo, el SDK oficial `docusign-esign`). Para sortear este error catastrófico de Build, el paquete problemático debe ser excluido explícitamente del bundling añadiéndolo al array `serverExternalPackages: ['docusign-esign']` en el archivo raíz `next.config.ts`, indicando al servidor que lo requiera de forma nativa en Node.js.
63. **Regla 63 (Preservación Activa de Potencias a Cero)**: El modal de edición de Leads y Contratos (`NewLeadModal.tsx`) NO debe sobreescribir las potencias (`p1c`-`p6c`) con valores a cero simplemente porque los campos de texto estén deshabilitados. La desactivación visual (`disabled={true}`) para evitar ediciones accidentales no debe impedir que el estado del componente mantenga y envíe el valor original de las potencias en el payload `contractData`.
64. **Regla 64 (Segregación Nativa de Direcciones y Titulares)**: Está PROHIBIDO usar expresiones regulares sucias en el Frontend para intentar recortar o extraer Códigos Postales, Poblaciones o Provincias de un campo de texto largo de `Calle`. El sistema extrae de forma nativa desde Airtable los campos individuales (`CP SOC`, `POBLACION SOC`, `PROVINCIA SOC`, `Tipo de vía Titular`, etc.) y los guarda limpiamente en sus propiedades correspondientes del `contractData`. Lo mismo aplica a los Nombres y Apellidos de Personas Físicas: el sistema detecta y retira limpiamente los apellidos del `businessName` para inyectarlos en `primerApellido` y `segundoApellido` sin usar chapuzas en el modal.
65. **Regla 65 (Traducción Inmediata a CRM Nativo en Duplicación)**: Al clonar o duplicar un Lead antiguo proveniente de Airtable, el Backend NO debe arrastrar la basura del JSON antiguo. El sistema está programado (en `duplicateLeadAction`) para interceptar el `airtableData`, mapearlo internamente en el servidor hacia columnas relacionales de Prisma y objetos estructurados (`contractData`), y generar un lead clonado con `airtableData: null`. Esto asegura que los leads clonados nazcan 100% purgados y nativos al nuevo esquema.
66. **Regla 66 (Desplegable Dinámico e Inteligente de Productos)**: La selección de Producto Comercial en el CRM jamás debe ser un campo de texto libre. Siempre es un `<select>` desplegable inteligente que se alimenta en tiempo real de la base de datos de `/api/products`. Debe incorporar reglas de filtrado de negocio: Ocultar productos cuya "Tarifa de Acceso" no coincida con el CUPS, ocultar productos de "Tarifa de Excedentes" si el usuario marca NO en Autoconsumo, e incluir un fallback visual (`"(Histórico)"`) si el contrato contiene un producto antiguo ya descatalogado.
67. **Regla 67 (Taxonomía Estricta de Tramitaciones)**: El campo "Tramitación a realizar" (`tipoTramitacion`) responde a un estándar fijo inalterable. Los únicos valores permitidos en cualquier modal, formulario o base de datos son: `Alta nueva`, `Cambio comercializadora sin cambios`, `Cambio comercializadora con cambios administrativos`, `Cambio comercializadora con cambios técnicos`, `Cambio comercializadora con cambios técnicos y administrativos`, `Modificación de datos administrativos`, `Modificación de datos técnicos` y `Modificación de datos administrativos y técnicos`. La opción de `Activación por desistimiento` se excluye de este flujo general.
68. **Regla 68 (Gestión Robusta de Productos Históricos vs Nativos)**: El campo `product` en la tabla `Lead` puede ser un ID moderno CUID (que empieza por `cm`) o un texto literal heredado de Airtable. Cualquier lógica de servidor o de acciones (ej. `page.tsx`, `contractActions.ts`) DEBE ser resiliente y manejar ambos casos: si el campo empieza por `cm`, buscará por ID (`findUnique`); de lo contrario, buscará por nombre exacto (`findFirst`), aplicando OBLIGATORIAMENTE un `.trim()` para purgar los espacios finales corruptos que venían de Airtable (ej. `"AED 1P BOE Socio "`).
69. **Regla 69 (Interfaz Interactiva para Productos en el Lead)**: En la vista de detalle de un Lead o Contrato, el campo "Producto Ofrecido" no debe ser estático. Debe renderizarse como un enlace interactivo que, al hacer clic, abra el `ProductModal` flotante hidratado con la ficha completa del producto (`dbProduct`), permitiendo a los comerciales o backoffice editar rápidamente precios, comisiones o características del producto sin perder el contexto ni navegar a la pestaña del Catálogo.
70. **Regla 70 (Integridad Relacional de las Distribuidoras)**: Se abandona el uso de diccionarios hardcodeados en el frontend para nombres de distribuidoras. Se debe utilizar la tabla nativa `Distributor` en Prisma. El backend mapeará automáticamente la distribuidora de un contrato o lead extrayendo el código REE de 4 dígitos desde la posición 2 del CUPS (`cups.substring(2, 6)`) y buscando ese `reeCode` en la base de datos para devolver el nombre oficial completo y formateado (ej. `[0031] EDISTRIBUCION REDES DIGITALES SL.`).
71. **Regla 71 (Integridad y Segregación de Datos Bancarios - IBAN)**: El IBAN activo y definitivo para la facturación de un suministro pertenece y reside única y exclusivamente en el modelo `SupplyPoint`. Sin embargo, durante el proceso de alta o renovación de un contrato (mientras está en "Borrador" o "Tramitación"), el nuevo IBAN solicitado por el cliente se almacena temporalmente en el modelo `Contract` (`Contract.iban`). Esta segregación es obligatoria para evitar sobreescribir y corromper el IBAN legítimo de un CUPS existente en caso de que el nuevo contrato sea rechazado.
72. **Regla 72 (Actualización Atómica del Punto de Suministro)**: Los datos técnicos de la instalación (Tarifa de Acceso, Potencias P1-P6) y los datos bancarios y administrativos (IBAN, CNAE) no deben sobrescribirse en el registro del Punto de Suministro (`SupplyPoint`) simplemente por la creación o edición de un contrato. La sincronización hacia el CUPS se realiza de forma **atómica y exclusiva** en el exacto momento en que el contrato transiciona a estado `ACTIVO`.
73. **Regla 73 (Validación de Congruencia de Productos y Tarifas)**: Durante el proceso de Alta de Luz o "Creación de Contrato", el backend debe validar de forma estricta que la Tarifa de Acceso asignada al Producto seleccionado coincida exactamente con la Tarifa técnica que tiene el suministro. Si el producto es de tipo `2.0TD` y el contrato se intenta tramitar para un suministro `3.0TD`, el sistema debe rechazar la operación lanzando una excepción de validación ("Incongruencia de producto/tarifa").
74. **Regla 74 (Validación Cruzada SIPS vs Petición Comercial)**: Dependiendo del tipo de "Tramitación a realizar", el sistema de validación variará su indulgencia:
    - Si la tramitación es **Cambio comercializadora sin cambios**, **Modificación de datos administrativos** o **Cambio comercializadora con cambios administrativos**, el backend exigirá que las Potencias (P1-P6) y la Tarifa de Acceso solicitadas en el contrato sean **exactamente iguales** a las obtenidas directamente de la base de datos de SIPS.
    - En caso de Alta Nueva, Modificación de Datos Técnicos, o Cambio con cambios técnicos, se permitirá la discrepancia técnica.
75. **Regla 75 (Validación del Representante Legal)**: Si en la creación o edición de un Lead/Contrato se estipula que el cliente es una entidad jurídica (CIF, Comunidad de Propietarios o Empresa), el backend debe requerir obligatoriamente que se informen de los datos del Representante / Apoderado (`DNI`, `Nombre` y `Apellidos`). Un contrato B2B no puede avanzar a firma sin un representante físico identificado.
76. **Regla 76 (Sobreescritura Progresiva de Datos de Contacto)**: A diferencia del IBAN y los datos técnicos (Regla 72), si un comercial genera un contrato para un cliente ya existente (`Client` activo) y proporciona nuevos números de teléfono o correos electrónicos (`email`, `phone`), estos datos de contacto SÍ deben sobreescribirse o añadirse de forma constructiva a la ficha general del `Client` de forma inmediata.
77. **Regla 77 (Resolución de Conflictos CUPS en Migración)**: Si un mismo CUPS aparece en varios contratos históricos de Airtable con diferentes IBANs, CNAEs o Datos Técnicos (potencias/tarifas), el script de importación y el CRM deben consolidar en la ficha principal del Punto de Suministro (`SupplyPoint`) únicamente los datos provenientes del contrato **más reciente** (aquel cuya fecha de firma o activación sea superior), asumiendo que esa es la última voluntad del cliente y el estado actual de la instalación.
78. **Regla 78 (Pertenencia Técnica del CNAE)**: A nivel arquitectónico y de base de datos, el código CNAE (Clasificación Nacional de Actividades Económicas) es un atributo intrínseco del Punto de Suministro (CUPS) y NO del Titular (`Client`). Un mismo titular jurídico puede poseer múltiples locales o naves con actividades económicas y CNAEs totalmente diferentes.
79. **Regla 79 (Jerarquía de Enriquecimiento SIPS)**: Las llamadas a la API de SIPS (Ingebau) sirven para validar y enriquecer la información técnica del CUPS. Si SIPS devuelve campos vacíos, el CRM jamás debe sobrescribir ni borrar los datos introducidos manualmente por el comercial. Sin embargo, **SIPS tiene jerarquía absoluta sobre el CNAE y el Código Postal de la instalación**. Si SIPS devuelve un CNAE o un Código Postal válidos, el CRM debe sobrescribir automáticamente lo que el comercial introdujo (inyectándolo en `contractData.cnae` y `contractData.direccionSuministro.postalCode`), ya que la Distribuidora rechazará cualquier alta que no coincida exactamente con sus registros en estos dos campos técnicos.
Tras la purga de campos innecesarios y basuras de Airtable, los campos que quedan definitivamente en la tabla `Lead` y que son los únicos permitidos y utilizados nativamente por el CRM o el script de importación son:

- **Core & Identidad**: `id`, `businessName`, `vatNumber`, `email`, `phone`, `status`, `source`, `createdAt`, `updatedAt`.
- **Datos SIPS / Técnicos Iniciales**: `cups`, `tariff`, `annualConsumption`, `estimatedMWh`.
- **Datos de Representante Legal**: `representativeName`, `representativeVat`.
- **Vínculos Estructurales**: `userId` (Comercial), `channelId` (Canal de venta), `brandId` (Marca), `contractId` (ID del contrato generado si se cierra con éxito).
- **Datos Json Inflexibles / Históricos**: `contractData` (Se guardan parámetros dinámicos no estructurales del FrontEnd o variables menores sin crear columnas Prisma), `sipsRawData` (volcado JSON puro de la respuesta técnica de Ingebau), `airtableId` y `airtableData` (Para rastreo 1 a 1 en las migraciones, sin ensuciar columnas).

**REGLA DE ORO DEL LEAD**: SALVO PETICIÓN EXPRESA, ESTÁ TERMINANTEMENTE PROHIBIDO CREAR NUEVOS CAMPOS EN LA TABLA `Lead` de `schema.prisma`. Cualquier dato temporal o accesorio irá en `contractData`.

## 57. Diccionario Definitivo de Datos en Prisma: Tabla CONTRACT
Tras la limpieza masiva de campos heredados de Airtable (93 fantasmas purgados), la tabla `Contract` de Prisma contiene estrictamente las columnas operativas del CRM:

- **Identificadores y Vínculos**: `id`, `contractCode` (Código visual), `version`, `previousContractId` (Trazabilidad), `clientId`, `supplyPointId`, `productId`, `userId`, `brandId`, `airtableId`.
- **Ciclo de Vida y Fechas**: `status` (BORRADOR, ACEPTADO, ACTIVO, etc.), `tramitationType`, `duration` / `permanenceMonths`, `createdAt`, `updatedAt`, `signatureDate`, `requestDate`, `activationDate`, `permanenceStartDate`, `expectedEndDate`, `terminationDate`.
- **Snapshot Económico Fijo (Foto de la venta)**:
  - `p1c` a `p6c` (Potencia contratada kW)
  - `p1e` a `p6e` (Precio energía firmado)
  - `p1p` a `p6p` (Precio potencia firmado)
1. **Borrador**: Si un Lead (Alta Luz) se envía a firma (manualmente o automáticamente), se genera el PDF del contrato y se crea un registro en la tabla `Contratos` con estado **Borrador**.
2. **Aceptado**: Cuando el cliente firma el contrato, el estado cambia a **Aceptado**. En este momento, se debe generar un registro en la sección de `Clientes` asignando a este nuevo cliente a la comercializadora correspondiente.
3. **Tramitando**: Cuando se inicia la tramitación del contrato con la distribuidora (mediante fichero switching), el contrato pasa a estado **Tramitando** (esto se podrá hacer manualmente o automáticamente vía scrapping futuro).

5. **Finalizado**: Cuando un contrato causa baja, el estado pasa a **Finalizado**.

## 6. Reactivación por Desistimiento (Tipo E1)
- Si un cliente, una vez finalizado su contrato por haberse ido a otra comercializadora, presenta un desistimiento y la distribuidora lo devuelve:
  - Se activa un proceso de reactivación tipo **E1**.
  - Se clona la versión del último contrato que estuvo activo.
  - Se **mantiene la fecha de inicio de permanencia inicial**.
  - Se asigna una **nueva fecha de alta** que coincidirá con la fecha en que la distribuidora comunique la reactivación oficial.

No simplificar ni eliminar campos sin autorización, ya que Make y automatizaciones externas pueden depender de ellos.

## 8. Lógica Avanzada de Contratos (Histórico)
- **Contratos Multipunto**: Hay clientes (CIFs específicos) que tienen un "contrato multipunto genérico". Para ellos, cualquier CUPS nuevo que dan de alta NO necesita generar PDF ni enviarse a firma. Entran directamente al sistema en estado **Aceptado** y se vinculan a dicho contrato.
- **Histórico de Contratos (Trazabilidad)**: Si se genera un contrato para un CUPS que ya tiene otro contrato activo previo, el nuevo contrato **debe almacenar el ID del contrato inmediatamente anterior** para poder arrastrar y calcular datos.
- **Cambios de Titularidad**:
  - *Subrogación*: Solo se genera un anexo de subrogación (una nueva "versión" del contrato actual) a firmar por el nuevo titular. No se activa hasta confirmación de distribuidora.
  - *Traspaso (Nuevo Contrato)*: Se genera un contrato nuevo desde cero. Cuando se activa, el contrato anterior causa baja.
- **Fecha de Inicio de Permanencia**: 
  - Coincide con la fecha de Alta en la Comercializadora **EXCEPTO** si el trámite es **M1** (Modificación de titular) o **E1** (Desistimiento).
  - Excepción a la excepción: Si se trata de un **M1S** (Modificación administrativa) con "versión vacía" (es decir, proviene de un nuevo contrato y no de una subrogación), la permanencia sí coincide con el alta.
- **Modificaciones Contractuales**: 
  - Las técnicas se gestionan como **M1N** (ej. cambio tarifa o potencia).
  - Las administrativas con subrogación se gestionan como **M1S**.

### 9.1 Lógica de Plantillas de Contrato (CORE vs B2B)
- Existen dos modalidades principales de plantillas de contrato: **CORE** y **B2B**.
- **SEPA B2B**: Solo aplicable a empresas y autónomos. En el mandato SEPA de esta plantilla, el cliente renuncia expresamente a su derecho de devolución del recibo (reembolso en 8 semanas).
- **SEPA CORE**: Para particulares (y algunas empresas si no se exige B2B). Mantiene el derecho estándar de devolución de 8 semanas.
- El motor nativo de generación de PDFs del CRM deberá ser capaz de seleccionar la plantilla adecuada en base a la naturaleza del cliente (`Tipo de persona` / `CNAE`) o a la selección explícita del comercial durante el "Alta Luz", inyectando las mismas variables que históricamente se inyectaban vía Make (`{{nombretit}}`, `{{iban}}`, `{{p1e}}`, etc.).

## 11. Regla de Migración y Mapeo de Datos
- **MIGRACIÓN AL 100% (REGLA DE ORO)**: Existe una regla inquebrantable y expresa de que **EL 100% DE LOS CAMPOS** de las tablas de Airtable (en especial `leads`, `contratos`, `facturas`, `instalaciones`, `productos`, `canales`) **tienen que venir cada vez que migremos información** de Airtable a PostgreSQL. 
- **Prohibido perder información**: No estamos aquí para purgar datos ni simplificarlos. La intención es profesionalizar el CRM, pero jamás eliminar información histórica.
- **Uso del campo JSON `airtableData`**: Para cumplir con la regla del 100% sin ensuciar la base de datos relacional, todo el registro bruto de Airtable debe insertarse en el campo `airtableData (JSON)` de las entidades principales. **ESTÁ ESTRICTAMENTE PROHIBIDO** crear campos "basura" o crudos de Airtable directamente en `schema.prisma`. Los campos nativos de Prisma deben mantenerse limpios y los scripts de migración deben mapear los datos relevantes (como NIF, consumos) a estas columnas tipadas. Este JSON sirve como backup absoluto.
- **Mapeo Exhaustivo de Diccionarios**: A pesar de tener el `airtableData`, TODOS los campos documentados en los diccionarios Excel (`diccionario_contratos.xlsx`, etc.) deben ser mapeados obligatoriamente a sus columnas relacionales en Prisma. No se pueden ignorar campos secundarios (como Segundos Apellidos, Teléfonos Extra, Facturas Papel) ni usar valores por defecto (`00000` o `Desconocida`) si el diccionario indica un campo de procedencia.
- **Identificación de Registros Históricos vs Nativos**: No se deben crear campos booleanos estilo `isImported`. Para discriminar en el futuro si un registro es heredado de Airtable o ha sido creado de forma nativa por los comerciales en el nuevo CRM, se debe evaluar la existencia de los campos `airtableId` y `airtableData`. Si son nulos, el registro es 100% nativo.
- **Comisiones de Productos de AED Energía**: En la importación de datos de AED Energía, todos los productos de tarifa `2.0TD` se importan con tipo de comisión "Tramos de Potencia". Los productos que **NO** son `2.0TD` se importan con tipo de comisión "% sobre Margen".
- **Gestión Inteligente de Basura Airtable:**
  - **Archivos y PDFs**: Cualquier campo de tipo archivo (CIE, DNI, Contrato Firmado, etc.) no se debe almacenar como simples URLs dispersas en el contrato. Se deben crear registros en la tabla global `Document` vinculados al registro padre.
  - **Mails de Acceso**: Los campos como `Email Comercial`, `Email Gerente`, `Email Sup Canal` usados históricamente en Airtable para dar permisos de vista a usuarios externos NO se importan. La seguridad en el nuevo CRM se gestiona por roles, marcas y comercializadoras nativas.
  - **Asignación de Comercial/Canal**: El identificador real de Comercial o Canal que venga de Airtable SÍ debe usarse para poblar las claves foráneas (`userId`, `channelId`) en los Contratos y Leads, de modo que hereden los permisos nativamente.
  - **Basura literal**: Se pueden ignorar de forma excepcional campos basura evidentes como `P1PM copy copy`, o fórmulas temporales para Make, siempre y cuando NO supongan información de negocio real.

## 14. Lecciones Aprendidas y Prevención de Errores (Edge Cases)
Para evitar tropezar dos veces con la misma piedra durante el desarrollo continuo del CRM, el desarrollador (IA) debe tener siempre presente los siguientes casos especiales resueltos:

1. **Relación Contract-SupplyPoint**: En Prisma, el modelo `Contract` tiene un único punto de suministro vinculado (`supplyPoint`, **en singular**), mientras que `Client` tiene múltiples (`supplyPoints`, **en plural**). Es crítico no usar `supplyPoints` en los `include` de Prisma cuando se consulta un contrato, ya que provocará un `PrismaClientValidationError`.
2. **Prioridad de Nombres y Trampas de Airtable**: Al importar clientes, es vital crear una función limpiadora (`trim()`) que descarte celdas que solo contengan espacios en blanco (ej. `" "`), ya que Airtable permite su inserción por error humano. La regla B2B/Físico es: Si `"Primer Apellido"` existe, se usa `"NOMBRE Y APELLIDOS"`. Si no, se asume Empresa y se usa `"NOMBRE/RAZON SOCIAL"`, con un fallback a `"NOMBRE Y APELLIDOS"` o `"Nombre completo Titular"` si la razón social falla.
3. **Manejo Defensivo de Arrays en Airtable**: Campos relacionales o de búsqueda (Lookups) como `"Comercial"` o `"CANAL"` viajan a través de la API de Airtable encapsulados en Arrays (ej: `["rec3fxKOf4kKpcbds"]` o `["Pablo Remacha"]`). **NUNCA** se debe asumir que son textos directos. Siempre hay que extraer la primera posición (`[0]`) y forzar un `String(...)` antes de renderizar en UI o aplicar métodos de texto como `.slice()` o `.toUpperCase()`, evitando así cuelgues del Front-End (`TypeError`).
4. **Mapeo Relacional de Comerciales (Fin del Diccionario Hardcodeado)**: Inicialmente se intentó usar un diccionario estático en código para traducir iniciales (`JP -> Javier Ponferrada`). Esto es una **mala práctica**. La regla oficial es que los comerciales se importan desde la tabla original `USUARIOS` de Airtable directamente a la tabla `User` de Prisma (filtrando solo los que tienen `CANALES LINK`, es decir, los verdaderos comerciales). Para saber qué comercial vendió un contrato/lead, se busca en Prisma el usuario cuyo `email` coincida con el array `airtableData['Email Comercial'][0]`, y se vincula relacionalmente (`lead.userId = user.id`). Así, la UI simplemente renderiza `lead.user.name`.
5. **CUPS MOCK y Validaciones SIPS (Ingebau)**: Durante las importaciones, si un contrato/lead carece de CUPS temporalmente, los scripts inyectan un identificador `CUPS_MOCK_...` para no romper relaciones nulas en Prisma. Sin embargo, al consultar a SIPS (Ingebau), el motor de back-end debe detectar estos MOCKS, ignorarlos y tratar de escarbar en `airtableData['CUPS2']` antes de lanzar la petición. Ingebau colapsará si recibe un CUPS MOCK.
6. **Interpretación de Potencias (kW vs W)**: En Airtable existen los campos `P1C`, `P2C`, etc. que contienen la potencia en kW (con separadores de coma o punto). Existen campos paralelos llamados `P1C W` que contienen fórmulas que lo multiplican por 1000 para obtener Vatios. En nuestro software **debemos ignorar por completo las fórmulas en vatios (P1C W)** y utilizar SIEMPRE los campos originales `P1C`, parseando las comas por puntos, asumiendo su valor original en kW para evitar alteraciones o anomalías.
7. **Caché Persistente de Prisma en Next.js (Error SELECT *)**: Tras alterar el `schema.prisma` y eliminar columnas obsoletas (ej. `customFee`, `autoconsumo`), ejecutar `npx prisma generate` **NO es suficiente**. Next.js mantendrá en memoria caché la instancia antigua del cliente de Prisma. Al ejecutarse operaciones como `prisma.lead.findMany()` en cualquier página, Prisma intentará hacer un `SELECT *` solicitando la columna que ya no existe y lanzará errores `PrismaClientKnownRequestError` de tipo "The column X does not exist in the current database". Para solucionarlo en **todas** las páginas de golpe, es de obligado cumplimiento **detener la terminal (Ctrl+C) y arrancar en frío el proceso del servidor (`npm run dev`)** tras alteraciones destructivas en el esquema.
8. **Semántica de la Tabla USUARIOS de Airtable**: Al tratar con la tabla de usuarios heredada, se ha aprendido que: `Email Link` o `Email` coinciden con el email de acceso a Softr y será el login del CRM. `Nombre2` es el nombre real y completo del usuario comercial. `Código` es su identificador de comercial (ej. `PR`). `Supervisor canal` marca a los responsables de canal. El resto de emails y comisiones quedan relegados porque el CRM gestionará la seguridad por Roles y Marcas/Canales nativos.
9. **Semántica de la Tabla CANALES de Airtable**: De la tabla de canales heredada se extrae que la relación más crítica a mantener es la de `PRODUCTOS` (determina qué tarifas/productos puede visualizar y vender cada canal) y la de `COMERCIALES` (qué usuarios pertenecen a este canal). Además de su `Nombre` y `Código Canal`, existen emails para notificaciones (`Email Supervisor`, `Email Administracion`) que se mantendrán para envíos automáticos. Los campos de comisiones o automatismos genéricos (`% Comisión`, `GEN_CONTRATO_AUTO`) no se importan de forma dura, pues el CRM lo gestionará con su propio motor.
10. **Idempotencia y Purga en Migraciones**: Para evitar duplicidades de productos, clientes o facturas fantasmas, el script de migración SIEMPRE debe realizar un borrado en cascada (`deleteMany` o limpieza de base de datos) antes de insertar los nuevos datos, ya que no se pueden garantizar identificadores únicos estables que permitan hacer `upsert` seguro de forma masiva sin un UUID.
11. **Tipos de Comisión (POWER_TIERS vs MARGIN_PERCENTAGE)**: La forma en que se comisiona un contrato depende de la `Tarifa` del producto subyacente. Si la tarifa es `2.0TD` (Residencial/PYME), el tipo de comisión SIEMPRE debe fijarse internamente a `POWER_TIERS` (ramos por tramo de consumo, ej: <3000 kWh = 30€). Para cualquier otra tarifa (`3.0TD`, `6.1TD`, etc.), el tipo es `MARGIN_PERCENTAGE` (se paga un % sobre el Fee o margen industrial). Esta es una regla absoluta en el CRM que no debe depender de banderas manuales en Airtable.
12. **Pureza de Base de Datos y Prohibición de Joins Dinámicos para Precios Históricos**: Durante la migración, la base de datos PostgreSQL debe consolidar la "foto fija" del contrato (sus precios firmados, fees, excesos, etc.) directamente en los campos nativos de la tabla `Contract` y `Lead` (ej: `p1e`, `p1p`, `fee`, `pexc`). **Queda ESTRICTAMENTE PROHIBIDO heredar, calcular o leer precios dinámicamente haciendo join con el Catálogo de Productos (`Product`) en la UI o en los procesos de facturación.** La tabla de Productos es dinámica y sus precios cambian; los contratos históricos NO deben alterarse si cambia el catálogo. Cualquier referencia en el código a `product.p1e` o `customFee` en lugar de `contract.p1e` o `contract.fee` romperá la base de datos porque esas columnas "custom" obsoletas han sido purgadas del esquema de Prisma.
13. **Prioridad de Snapshots Estáticos frente a Lookups Dinámicos (Airtable)**: En Airtable, existen columnas que son lookups dinámicos (`P1E (from PRODUCTOS)`, `P1P (from PRODUCTOS)`) and columnas que son snapshots puros del momento de la firma (`P1P`, `P2P`, etc.). En la migración, se ha descubierto que Airtable SÍ guarda un snapshot estático independiente para los precios de **Potencia** (`P1P`), pero NO lo hace para la **Energía** (`P1E`). Por tanto, el script de migración SIEMPRE debe priorizar el snapshot puro para la potencia (`f['P1P'] || f['P1P (from PRODUCTOS)']`), para asegurar que prevalece el precio histórico (ej: 40.39€) firmado por el cliente en el contrato original, evitando que sea sobrescrito por el precio lookup vinculado al producto actual del catálogo.
14. **Regla 14 (Vinculación de Facturas por Contrato Airtable)**: Las facturas en Airtable están vinculadas al contrato a través de una columna Lookup Múltiple en `Contratos` (llamada `FACTURASfldaa54Nua6LxjjX7`). La importación de facturas DEBE recorrer esta matriz de IDs y hacer *fetch* individual por ID para evitar los límites de paginación (`maxRecords`) y asegurar que se descarga el "árbol completo" de facturas de un contrato.
15. **Regla 15 (Capitalización en Prisma)**: Cualquier inclusión relacional en el código Prisma (`include: {}`) debe hacerse estrictamente en *camelCase* (`contract`, `supplyPoint`, `invoice`, `product`, `lead`), de acuerdo a la configuración del esquema. El uso de PascalCase en los includes provocará `PrismaClientValidationError`.
16. **Regla 16 (Datos del Representante/Apoderado en Airtable)**: Según el diccionario de datos de Airtable, para los contratos B2B (Personas Jurídicas), los campos `Nombre Contacto`, `Apellidos Contacto` y `NIF Contacto` almacenan en realidad los datos del Representante Legal / Apoderado. El script de migración mapea estos campos directamente a `representativeName`, `representativeVat`, etc., para que no aparezcan vacíos.
17. **Regla 17 (Nomenclatura y Mapeo de Conceptos SVA)**: El campo `Servicio` en Airtable devuelve un ID de registro interno (`rec...`). Este ID debe ser mapeado explícitamente a su nombre real (ej. "Alta Nueva (25 €/alta)") mediante un diccionario estático en la migración y el frontend, en lugar de intentar extraerlo del campo fórmula `Producto y Servicio` con expresiones regulares, ya que este último a menudo no incluye comillas.
18. **Regla 18 (Tipos de Producto)**: En Airtable, el `Tipo de producto` es una cadena de texto explícita (ej. "Precio indexado Pass-Through", "Precio fijo único"). En lugar de reducirlos a constantes binarias (`FIX` o `INDEXED`), Prisma y la UI almacenan y renderizan la cadena literal exacta, garantizando mucha más claridad a nivel comercial.
19. **Regla 19 (Paginación Oculta de Airtable)**: La API de Airtable devuelve un máximo estricto de 100 registros por consulta. Queda TERMINANTEMENTE PROHIBIDO realizar descargas directas sin paginación para tablas maestras (`PRODUCTOS`, `CANALES`, `USUARIOS`, etc.). El script de migración siempre debe implementar un bucle `do-while` que inyecte el parámetro `offset` en la URL hasta que ya no haya más páginas, evitando así truncamientos catastróficos de catálogos (ej. perder las tarifas 2.0TD por estar a partir de la fila 100).
20. **Regla 20 (Limpieza de Espacios y Productos Fantasma)**: Los usuarios de Airtable introducían espacios finales por error (ej. `"AED 24h "`). Para evitar que el CRM falle al buscar el producto al importar un contrato, la regla de oro es aplicar siempre `.trim()` a los nombres de productos antes de cualquier `findFirst`. Además, si un producto referenciado en un contrato realmente no existe, el CRM debe crear un "Producto Fantasma" para evitar orfandad, al cual **se le debe asignar obligatoriamente la Tarifa** que venga en la fila de dicho contrato (`f['Tarifa']`) y su tipo de comisión correspondiente, en lugar de dejar la tarifa a `null`.
21. **Regla 21 (Separación de Puntos de Suministro por Titular / Prevención de Fugas de Datos)**: Es absolutamente CRÍTICO entender que un `SupplyPoint` no es solo una entidad física (CUPS), sino una entidad física **asociada a un cliente** (por eso existe el constraint `@@unique([cups, clientId])`). El `SupplyPoint` almacena datos sensibles como el `IBAN` o el `CNAE` del titular en ese momento. Cuando se realiza un cambio de titular en un CUPS existente (o un alta nueva de otro titular), **ESTÁ TERMINANTEMENTE PROHIBIDO hacer un `prisma.supplyPoint.update` que sobrescriba el `clientId` y el IBAN del registro anterior**. Hacer esto genera una fuga de datos (GDPR) donde el antiguo titular, al ver su histórico, visualizará el IBAN del nuevo titular. La regla de oro es: si un contrato cambia de titular, el CRM **debe clonar y crear un nuevo registro de `SupplyPoint`** para el nuevo `clientId` conservando los datos físicos, pero manteniendo los datos bancarios separados y exclusivos para cada cliente. Cada cliente tiene su propia fila de `SupplyPoint` aunque compartan CUPS.
22. **Regla 22 (Campos Obsoletos del UI y Nuevo Motor de Comisiones)**: Las columnas históricas de `IP` (Indexado Potencia), `FC` (Fijo Comercial) y `Costes Estimados (P1E_Coste...)` han sido purgadas del `schema.prisma` debido al "Nuevo motor de comisiones". El Frontend NO DEBE renderizar "campos fantasma" en los formularios (ej. `ProductModal.tsx`) que intenten enviar estos datos al Backend, ya que causará un Error 500 en Prisma por columna inexistente. Solo se renderizarán y editarán campos reales de BD.
23. **Regla 23 (Fechas de Facturación - Legacy vs DateTime)**: El esquema de Prisma para `Invoice` contiene campos de fecha fuertes (`billingStart`, `billingEnd`) y campos de cadena legacy de Airtable (`desde`, `hasta`, `desdeEA`, `hastaEA`). Para que la visualización UI (ej. "Periodo Energía") legacy no se rompa o muestre "-", el script de migración debe rellenar ambos tipos de campos simultáneamente.
24. **Regla 24 (SuperAdmin Global)**: El usuario super administrador por defecto del CRM es `fjponferrada@sp-energia.com`. Al realizar migraciones, los contratos y leads no asignados explícitamente se volcarán bajo este usuario en lugar de cuentas ficticias (como `admin@migracion.com`), garantizando así que los datos no se queden "huérfanos".
24. **Regla 24 (Migración y Asignación de Canales)**: Airtable almacena los canales de distribución en una tabla independiente (`CANALES`). La importación de datos descarga estos canales y los inserta en el modelo `Channel` (usando el Record ID de Airtable como clave primaria `id` y `code`). Al procesar un contrato, si se detecta que pertenece a un canal y tiene un comercial asociado (`Email Comercial`), el CRM vinculará automáticamente el usuario a su respectivo `channelId`, permitiendo que la UI refleje el canal del agente comercial con claridad.
25. **Regla 25 (Catálogo de Productos por Canal)**: Los Canales tienen una relación Muchos a Muchos (`Many-to-Many`) con los Productos. Un canal solo tiene permiso para vender los productos que se le hayan habilitado en su matriz `PRODUCTOS`. En la migración, se debe cruzar el array de IDs original con los Productos ya importados, y en la UI, el Modal de Canal debe listar explícitamente el catálogo autorizado para dicho canal.
26. **Regla 26 (Limpieza e Importación de Usuarios/Comerciales)**: En la migración de datos, se purga la tabla de `User` (a excepción estricta del Superadmin `fjponferrada@sp-energia.com` o cuentas críticas de sistema) para luego importar masivamente desde la tabla `USUARIOS` de Airtable. **Filtro activo**: solo se traen aquellos registros que disponen de `CANALES LINK` (los verdaderos comerciales en activo). **Asignación de Roles automática**: si el usuario tiene marcado el checkbox `Supervisor canal` (`true`) en Airtable, se le da el rol de `CANAL` (`isChannelSupervisor = true`). De lo contrario, se le asume agente raso y recibe el rol de `COMERCIAL`.
27. **Regla 27 (Seguridad Estricta por Marcas Asignadas)**: Aunque el esquema de Prisma obliga a que cada usuario tenga una `brandId` como marca primaria o por defecto, **el acceso real al sistema está dictado EXCLUSIVAMENTE por sus Marcas Asignadas (`assignedBrands`)**. Si a un usuario (CANAL o COMERCIAL) se le desmarcan todas las marcas autorizadas en su ficha, el sistema arrojará un error 403 / "Acceso denegado" en el Login, ignorando la marca primaria subyacente. Para mantener la coherencia en usuarios migrados antiguamente (que solo tenían `brandId`), un script inicial de "backfill" copió su marca primaria dentro de su array de `assignedBrands`.
28. **Regla 28 (Jerarquía de Permisos Visuales: Companies -> Brands)**: En la interfaz (UI) y a nivel lógico, el acceso a una marca (`assignedBrands`) es contingente a tener acceso a su comercializadora padre (`companies`). Si a un usuario no se le asigna la entidad matriz en su relación `companies`, la UI no renderizará las marcas dependientes de la misma, garantizando que los permisos se otorgan siempre de forma jerárquica de arriba abajo.
29. **Regla 29 (Campos no estructurados - `contractData` en Leads)**: El esquema relacional Prisma para el modelo `Lead` debe mantenerse limpio de campos accesorios, temporales o inestables (ej. `titularStreetType`, `contactRole`, etc. que vienen de formularios dinámicos). Todo campo que no pertenezca estrictamente al modelo CORE relacional del Lead debe anidarse de forma segura dentro del campo JSON `contractData`.
30. **Regla 30 (Validación de Tipos Prisma en Next.js)**: Cualquier alteración estructural al `schema.prisma` (renombrar relaciones, eliminar columnas) requiere no solo `npx prisma generate`, sino que es de obligado cumplimiento matar los procesos y limpiar la caché de compilación de Next.js. De lo contrario, Turbopack utilizará la declaración de tipos obsoleta en caché y el build fracasará con un `PrismaClientValidationError`.
31. **Regla 31 (Pureza de API en Endpoints)**: Los endpoints de la API (ej. `/api/products`) deben estar estrictamente saneados. Propiedades antiguas o variables de configuración legacy que ya no existan en el esquema oficial de Prisma (como `ip`, `fc`, `gasIncluido`) NO pueden ser inyectadas en las cláusulas `data: {}` de creación/actualización. Hacerlo provocará un error catastrófico de tipado *"Object literal may only specify known properties"* durante la compilación.
32. **Regla 32 (Aislamiento Estricto Multi-Marca)**: Para garantizar el funcionamiento como SaaS Multitenant y el cumplimiento de RGPD, los identificadores como `vatNumber` (DNI), `cups` o `invoiceNumber` **NO SON GLOBALES**. Su unicidad está delegada y combinada estrictamente con el `brandId`. Un mismo DNI o CUPS puede existir múltiples veces en la base de datos siempre que pertenezcan a marcas diferentes. Todo el backend (vía `findFirst`), listados UI (vía `VisibilityFilters`) y operaciones CRUD deben inyectar ineludiblemente el `brandId` del usuario o sesión actual para garantizar que jamás haya fuga o cruce de datos entre comercializadoras distintas.
33. **Regla 33 (Única Instancia de Prisma en Next.js/Vercel)**: En entornos Serverless como Vercel, está terminantemente prohibido instanciar `new PrismaClient()` manualmente dentro de los Server Components (`page.tsx`) o endpoints de la API. Hacerlo agotará el *connection pool* de la base de datos y causará errores `PrismaClientKnownRequestError` o fallos de lectura de la variable `DATABASE_URL` frente a `PRISMA_DATABASE_URL` / `POSTGRES_URL`. Todos los archivos deben inyectar ineludiblemente la instancia global y cacheada de Prisma importándola exclusivamente desde el archivo central: `import { prisma } from '@/lib/prisma'`.
34. **Regla 34 (Contexto de Migración Airtable -> AED)**: Todo el contenido y registro histórico proveniente de los backups de Airtable pertenece 100% en exclusiva a la comercializadora semilla "AED Energía" y a su marca homónima. Al ejecutar scripts de migración, **ESTÁ PROHIBIDO** purgar o borrar las tablas de `Company` o `Brand`. El script de importación debe buscar la marca "AED" existente en la base de datos y vincular relacionalmente el 100% de los Usuarios, Clientes, Contratos y Leads importados a dicha marca.
35. **Regla 35 (Gestión Autónoma del Servidor de Desarrollo)**: Cada vez que el servidor local de desarrollo (`npm run dev`) se caiga, el ordenador se reinicie, o el usuario pida explícitamente "arrancar/reiniciar el servidor", el agente de IA **debe** encargarse directamente de lanzar el comando de ejecución en segundo plano sin delegarlo en el usuario.
36. **Regla 36 (Fusión de Oportunidades y Leads Airtable en CRM SP)**: 
    - **Flujo original en AED (Airtable)**: Se usaban dos tablas distintas. `OPORTUNIDADES` (datos básicos para lanzar ofertas) y `LEADS` (si aceptaba la oferta, se pedían datos completos para generar borrador de contrato PDF). Si lo firmaba, pasaba a `CONTRATOS`. Luego las facturas se ataban al contrato. Relación estricta: Lead -> Contrato -> Facturas.
    - **Flujo nuevo en CRM SP Energía**: Ambas fases (oportunidad rápida y borrador de contrato completo) se fusionan y unifican bajo la tabla y sección única de `Leads`. Desde esta única página, el usuario puede lanzar una oferta básica ("Crear Oferta") o tramitar el contrato completo ("Alta Luz") como indica la Regla 4. Al migrar los datos de Airtable, se debe tener en cuenta esta fusión unificadora.
37. **Regla 37 (Aislamiento Multimarca de Canales)**: Históricamente concebidos como entidades globales, los Canales (`model Channel`) están estrictamente aislados y pertenecen a una única marca (`brandId`). El backend, las vistas (UI) y las consultas Prisma deben filtrar rigurosamente por el `brandId` del usuario activo (`getChannelVisibilityFilter()`) para evitar que responsables de distintas comercializadoras vean canales de su competencia. Las migraciones y creaciones de nuevos canales inyectarán siempre el `brandId` adecuado.
38. **Regla 38 (Estrategia de Migración y Sincronización Incremental)**: El proceso de volcado masivo de datos desde Airtable se divide en dos enfoques para garantizar seguridad, integridad y tolerancia a fallos:
    - **Migración Masiva (`migrate-airtable.ts`)**: Proceso destructivo de "borrón y cuenta nueva". Elimina toda la base de datos (excepto administradores clave) e importa el 100% de la jerarquía (Compañías, Marcas, Canales, Productos, Usuarios, Contratos, Leads y Facturas). Las facturas se precargan en memoria (un solo fetch global) para evitar el colapso de la API por peticiones N+1. Cualquier error individual en un contrato se atrapa de forma silenciosa y se vuelca en un log persistente (`migration_errors.json`).
    - **Sincronización Incremental (`sync-missing.ts`)**: Proceso no destructivo de "recuperación". Diseñado para reparar o importar contratos faltantes tras resolver errores del primer volcado. Compara los `airtableId` de la BD local con los de Airtable, detecta el delta (los ausentes) y procesa única y exclusivamente esos registros sin borrar nada del entorno productivo, generando su propio log `sync_errors.json` en caso de fallos. Todas las búsquedas previas de CUPS, DNI (`vatNumber`) o número de factura deben incluir ineludiblemente el contexto de la marca (`brandId` o `clientId`) como dicta la Regla 32.
39. **Regla 39 (Prioridad del CIF Original y Mapeo Completo del Cliente)**: El verdadero identificador fiscal en los contratos históricos se extrae del campo de texto `CIF`, no de columnas derivadas o links como `Copia de CIF link`. Al migrar contratos, si el cliente (Client) no existe en BD por coincidencia estricta de `vatNumber` y `brandId`, se debe crear la nueva ficha absorbiendo el 100% de la información comercial proveniente del registro de Airtable del contrato (incluyendo `DOMICILIO SOC`, `CP SOC`, `POBLACION SOC`, `IBAN`, `CNAE`, etc.). Jamás se deben dejar fichas de clientes huérfanas de información por crear un cliente de emergencia sin DNI.
40. **Regla 40 (Preservación Detallada de Estados de Rechazo)**: Los estados de "Rechazado" en Airtable incluyen casuísticas que deben ser rastreables. El importador NO debe colapsar todos los estados de rechazo en un macroestado global `RECHAZADO`. Se deben generar y mantener los subestados específicos (`RECHAZO_DISTRIBUIDORA`, `RECHAZO_COMERCIALIZADORA`, `RECHAZO_RIESGOS`, `RECHAZADO_POR_CLIENTE`) en el campo `status`, dotando a la UI y a los filtros de granularidad total para la toma de decisiones.
41. **Regla 41 (Mapeo Estricto de Observaciones Contractuales)**: Toda anotación en el campo `Observaciones` (u `OBSERVACIONES`) de Airtable para un contrato debe volcarse incondicionalmente en la propiedad `internalComments` del modelo `Contract` en Prisma. Esta información es crítica y el Front-End está programado para renderizarla a modo de tooltip interactivo (`MessageSquare`) sobre el propio listado de contratos, sin necesidad de entrar a la ficha detalle.
42. **Regla 42 (Paginación SSR y Optimización de Consultas)**: Debido a que la base de datos supera holgadamente los miles de registros, queda terminantemente prohibido volcar o renderizar todas las filas de golpe (`findMany()` sin límites) hacia el cliente de React (UI). Todas las tablas maestras (Clientes, Contratos, Facturas) deben emplear Server Actions nativas con delegación de paginación a nivel de SQL (`skip` / `take`). Las estadísticas generales superiores (ej: Total Activos, MWh total) deben resolverse mediante consultas agregadas independientes para desacoplar el rendimiento de la tabla visual de la carga de cálculo matemático.
43. **Regla 43 (Cálculo Dinámico de Márgenes en Facturas)**: Para el dashboard y analítica económica, se ha integrado la fórmula de cálculo de márgenes reales (Margen Ingebau vs Margen Estimado). Es fundamental que el campo `margin` se calcule y consolide nativamente en la base de datos para cada factura importada, saneando valores anómalos.
44. **Regla 44 (Prevalencia del Fee de Contrato sobre el Producto)**: Al calcular márgenes o comisiones de una factura, el sistema SIEMPRE debe priorizar el `fee` extraído de la tabla `Contract` asociada a esa factura. El `fee` genérico del catálogo de productos (`Product`) solo se usará si el contrato no tuviese uno definido. Esto previene que alteraciones modernas en el precio base del catálogo corrompan la rentabilidad real que se le firmó al cliente en su día.
45. **Regla 45 (Cifra de Negocio y Abonos en el Análisis Económico)**: Para representar fielmente la "Facturación" o volumen de ventas en el Dashboard, JAMÁS se usará el campo `totalAmount` (que incluye impuestos). La facturación oficial es siempre la suma de la base imponible (`baseImponibleIva`). Además, el sistema invierte automáticamente el signo (haciéndolo negativo) de las bases imponibles si el `Tipo Factura` detectado es un `Abono`, asegurando la precisión contable de la cifra de negocio mensual.
46. **Regla 46 (Unificación del Panel de Control)**: Toda la analítica de negocio (Evolución de Cartera MWh, Altas, Bajas, Facturación vs Margen y Rendimiento) se debe condensar siempre en la pantalla principal del `Dashboard`. La política de interfaz exige evitar la fragmentación en sub-páginas analíticas para ofrecer al perfil directivo una visión global inmediata en portada.
47. **Regla 47 (Lógica de Falsas Bajas y Periodo de Gracia)**: A efectos de cálculo de KPI en el Dashboard y renderizado en la tabla principal de Contratos, existe la figura de la "Falsa Baja". Si un contrato finaliza (`status = 'BAJA'`), pero para ese mismo Punto de Suministro (CUPS) se activa otro contrato posterior en un plazo inferior a 30 días (Periodo de Gracia), se considera una continuidad comercial (ej. renovación o modificación) y NO una fuga de cliente. Estas "Falsas Bajas" se excluyen matemáticamente del recuento de churn del dashboard y se ocultan visualmente de los listados para no contaminar la operativa del comercial.
48. **Regla 48 (Histórico Estricto de Potencias Contratadas)**: En la misma línea que la Regla 12 sobre la preservación de precios de energía estáticos, las potencias contratadas en el momento de la firma (P1 a P6 en kW) deben almacenarse como un "snapshot" estático dentro del propio modelo `Contract` (`p1c`, `p2c`... `p6c`). Aunque un Punto de Suministro subyacente cambie su potencia actual con el tiempo (por una subida de potencia, por ejemplo), el registro histórico del contrato debe mostrar inalterable la potencia exacta con la que fue generado o importado originariamente.
49. **Regla 49 (Compatibilidad de Prisma Accelerate y Adaptador PG)**: En entornos Serverless de Vercel, si se utiliza el adaptador nativo `@prisma/adapter-pg` en combinación con el módulo de Node `pg` (para hacer pooling interno), **queda estrictamente prohibido** utilizar enlaces de Prisma Accelerate (`prisma+postgres://...`). La librería `pg` no soporta proxies Edge por el puerto 5432 y genera un `ETIMEDOUT` masivo. En esta configuración, Vercel debe conectarse SIEMPRE mediante el enlace directo (`postgres://...`). Para poder usar Accelerate, es requisito indispensable eliminar el adaptador `pg` y migrar a la extensión oficial `@prisma/extension-accelerate`.
50. **Regla 50 (Latencia Geográfica Vercel vs Prisma)**: Las funciones Serverless de Vercel se despliegan por defecto en Washington D.C. (`iad1`). Si la base de datos de Prisma Postgres está en Europa (ej. París `eu-west-3` o Frankfurt `eu-central-1`), el viaje transatlántico de los datos añadirá 1-2 segundos de "lag" en cada navegación de la interfaz. Para lograr tiempos de carga instantáneos (5-10 ms), **es de obligado cumplimiento** asegurar que la `Function Region` en el panel de Vercel coincida geográficamente con la región del Cloud Provider de Prisma (ej. Vercel `cdg1` para París o `fra1` para Frankfurt).
51. **Regla 51 (Límites de Facturación y Bloqueos Prisma Postgres)**: El plan gratuito (Free) de un Workspace en Prisma Cloud tiene un límite estricto de 100.000 consultas/operaciones al mes. Importaciones masivas de datos (como volcados SQL de Airtable de +1GB) sobrepasarán este límite en cuestión de minutos, provocando que la plataforma bloquee el acceso a la base de datos y la UI lance errores persistentes del tipo `Invalid 'prisma.entity.findMany()' invocation`. Toda migración histórica debe ejecutarse en bases de datos que cuenten con un plan de pago activo (ej. Starter 10$/mes) aplicado correctamente sobre el Workspace destino.
52. **Regla 52 (Almacenamiento en la Nube de Archivos y PDFs - Cloudflare R2)**: Para el alojamiento físico de documentos pesados (PDFs de facturas, contratos firmados, DNI, CIEs, etc.), el sistema utiliza como infraestructura base **Cloudflare R2** (un clon de Amazon S3 sin costes de salida de red). Queda estrictamente prohibido guardar archivos codificados en Base64 en la base de datos relacional de Prisma Postgres. El flujo de subida de cualquier archivo debe enviarse al endpoint `/api/upload`, el cual lo inyectará en Cloudflare R2 y devolverá una URL pública (`https://pub-fca...`). Esta URL es lo único que se almacena en PostgreSQL (generalmente mediante el modelo global `Document` o campos nativos como `pdfUrl`).
53. **Regla 53 (Procesamiento Nativo de Ficheros SCTD y Prevención de Sobreescritura)**: La ingesta de archivos XML de distribuidora (procesos C1, M1, A3, etc.) se hace de forma 100% nativa en el CRM para eliminar dependencias externas (Make/Drive). El archivo se procesa en memoria y se envía directamente a Cloudflare R2 para auditoría. Además, el parser incluye un sistema estricto de Warnings: Si un XML trae una fecha de activación para un CUPS, pero el contrato en la base de datos ya tiene ese campo de fecha lleno, el sistema **no sobreescribirá** la fecha para evitar daños por duplicidad, y en su lugar generará un registro de "Warning" para que el Backoffice lo revise manualmente en la página de Switching Warnings.
54. **Regla 54 (Fuente de Verdad para Estructuras XML CNMC)**: El parser de archivos XML no se diseña en base a suposiciones ni scripts antiguos, sino que debe programarse basándose estrictamente en los esquemas XSD oficiales publicados por la CNMC (Sistema de Comunicación de Datos o SCTD). Toda la documentación técnica, PDFs de los formatos (A3, C1, B1, etc.) y los 91 archivos `.xsd` oficiales se encuentran alojados y deben ser consultados en la ruta de red local: `Z:\AED\Switching\CNMC - E - V3.0 2024.05.16`.
55. **Regla 55 (Consistencia de Estados del Ciclo de Vida y Tipado Prisma)**: El estado final de un contrato cuando causa baja es estrictamente `Finalizado`, NUNCA `BAJA`. Las actualizaciones de estado en el backend (ej. al ingerir archivos XML de distribuidora) deben usar `Finalizado`. Además, las alteraciones o inyecciones de fecha deben apuntar siempre a las variables reales del esquema de Prisma (ej. `terminationDate` en lugar del antiguo `endDate`), ya que Prisma interceptará cualquier propiedad desconocida y abortará la transacción entera devolviendo un error 500, dando la falsa impresión de que un proceso (como la ingesta de archivos) ha funcionado a medias si el error es capturado silenciosamente.
56. **Regla 56 (Arquitectura Desacoplada para Facturas F1)**: Los archivos de facturación de peajes (proceso F1 de la CNMC) NO deben estar ligados estáticamente a un `contractId` en el momento de su ingesta. Deben asociarse únicamente al `supplyPointId` (CUPS). La relación entre un F1 y el contrato o comercializadora responsable se calculará **dinámicamente en tiempo de lectura** (tanto en la vista general como en la ficha del contrato). Este cálculo se realiza cruzando el periodo de facturación (`fechaInicio` y `fechaFin` del XML F1) con el periodo de vigencia del contrato (`activationDate` a `terminationDate`), previniendo que modificaciones retroactivas de contratos dejen facturas F1 secuestradas en contratos inactivos.
57. **Regla 57 (Protección de Ingesta y Exclusión de Procesos Cruzados)**: Para evitar colapsos en el parseo y contaminación de datos en la base de datos de eventos de Switching, el motor de ingesta de "Switching Warnings" bloqueará estrictamente cualquier XML cuyo `<CodigoDeProceso>` pertenezca a `F1` (y viceversa, el módulo de F1 bloqueará procesos que no sean F1). Además de basarse en la raíz del XML (ej. `<MensajeFacturacion>`), el chequeo en profundidad de las etiquetas de cabecera previene vulnerabilidades de inserción accidental por parte del usuario final.
58. **Regla 58 (Búsqueda de CUPS en Contratos Históricos Airtable)**: Los contratos y leads heredados de Airtable se volcaron enteros en el campo `airtableData`, por lo que el CUPS no siempre está en el modelo relacional nativo (`SupplyPoint`) si no se migró de forma perfecta en su día. Para evitar errores de "CUPS no encontrado" en procesos automáticos (ej. importar XML de Switching), el motor de búsqueda SIEMPRE debe realizar un fallback buscando el `codigoSolicitud` u ojeando el JSON `airtableData` en la tabla `Lead` si la búsqueda directa por la tabla `SupplyPoint` falla. Además, el CUPS de los ficheros de la CNMC puede llegar con 22 caracteres (incluyendo sufijos como `0F` o `1F`), por lo que las búsquedas en base de datos deben limitarse siempre a los primeros 20 caracteres del CUPS (`cups.substring(0, 20)`).
59. **Regla 59 (Procesos de Baja y Reposición CNMC - Diferenciación B1, B2 y E2)**: Existen tres flujos que causan confusión y que deben ser tratados en el importador de forma diferenciada para evitar cierres prematuros de contratos o errores de estado `CONTRATO_NO_TRAMITANDO`:
    - **B1 (Baja por Impago)**: En su paso `02` (Aceptación), debe marcar nativamente el contrato con `suspendido = true`. En su paso `05` (Activación Baja), cambia el estado del contrato a `Finalizado`.
    - **B2 (Baja a iniciativa del distribuidor)**: Según el estándar CNMC, el B2 consta de un único mensaje (paso `05`). Cuando llega, debe forzar el contrato a estado `Finalizado`. (Nota histórica: Internamente el equipo comercial a veces llama "B2" a las reposiciones, pero a nivel código se debe tratar estrictamente como baja).
    - **E2 (Reposición de Suministro)**: Es el verdadero proceso de reconexión (archivos como `ActivacionReposicion.xsd`). Cuando se procesan archivos E2 (pasos 14, 15, etc.), el sistema simplemente debe enlazarlos al contrato en estado `ACTIVO` sin alterar su estado ni la marca de `suspendido`.
60. **Regla 60 (Almacenamiento Físico de Contratos y Organización en Cloudflare R2)**: En base a la Regla 52, Vercel Blob ha sido descartado por completo para este ERP. Todo documento generado (borradores PDF o contratos firmados por DocuSign) se almacena obligatoriamente en Cloudflare R2. Queda estrictamente prohibido crear una "carpeta por ID de contrato". Para mantener el bucket limpio, los borradores se guardarán en plano en `contracts/drafts/Contrato_AED_{CodigoContrato}.pdf` y los firmados en `contracts/signed/contrato_firmado_{CodigoContrato}.pdf`.

61. **Regla 61 (Desnormalización para Búsqueda - "Computed Display Name")**: El campo `businessName` en la base de datos (tanto para `Lead` como `Client` y `Contract`) es un campo desnormalizado por diseño. Aunque existen columnas aisladas para guardar los apellidos (como `contractData.primerApellido` y `contractData.segundoApellido`), el sistema fuerza a que `businessName` contenga el "Nombre + Apellido 1 + Apellido 2" completo concatenado (en el caso de personas físicas). Esta redundancia es de obligado cumplimiento para evitar realizar operaciones de Full-Text Search multi-columna cuando el usuario filtra clientes en los listados del CRM.

62. **Regla 62 (Inyección Inteligente de Apellidos en Plantillas de Contrato)**: Al generar un borrador PDF de contrato, el sistema inyecta el nombre sin los apellidos en la variable `{{nombretit}}` y los apellidos en `{{1apetit}}` y `{{2apetit}}`. Dado que `businessName` (Regla 61) contiene el nombre entero, el inyector (backend) implementa una lógica de limpieza (elimina `1apetit` y `2apetit` de la cadena `businessName`) para extraer únicamente el Nombre Puro antes de rellenar la plantilla `CORE.docx` o `B2B.docx`. El inyector también debe parsear defensivamente campos complejos (como direcciones que puedan venir de importaciones previas en formato de objeto JSON) usando comprobación de tipos segura (`typeof field === 'object'`).

63. **Regla 63 (Comportamiento de Duplicación de Leads)**: La acción de "Duplicar Lead" no genera un clon idéntico ciego. Genera un clon pero le aplica un reseteo de estado para empezar un trámite desde cero: fuerza el estado a `NUEVO`, destruye los vínculos `contractId` y `docusignEnvelopeId` (para no solapar contratos antiguos), y purga el JSON importado de `airtableData`, traduciendo toda su información al esquema estructurado limpio del CRM nativo a través del parser oficial (`translateAirtableToNativeCRM`).

64. **Regla 64 (Prioridad del Estado Transitorio sobre el Consolidado para la Oferta/Contrato)**: En la generación de borradores de contrato (PDF y envío a firma), el sistema JAMÁS debe leer la dirección, IBAN o teléfono desde las tablas consolidadas del sistema (`SupplyPoint` o `Client`) de forma prioritaria, porque estos datos podrían ser históricos o estar atados a un contrato anterior. La jerarquía de extracción inquebrantable debe buscar primero en la "bolsa transitoria" de la negociación activa (es decir, en `contract.Lead` y su JSON asociado `airtableData`), recurriendo a `Client`/`SupplyPoint` únicamente como último y lejano fallback si el dato transitorio no existe.

65. **Regla 65 (Prohibición del SDK de DocuSign y Bug de Turbopack AMD)**: La librería oficial de Node.js de DocuSign (`docusign-esign`) presenta dos fallos críticos insalvables: omite parámetros esenciales del Payload (como `additionalNotifications` para SMS multicanal) y utiliza un formato de empaquetado antiguo (AMD `define()`) que revienta silenciosamente la fase de compilación (Build) del compilador ultrarrápido Turbopack de Next.js en Vercel. Por tanto, queda **estrictamente prohibida** su instalación. La autenticación JWT debe implementarse manualmente con criptografía nativa (librería `jsonwebtoken`) y la creación de sobres (`createEnvelope`) debe ejecutarse mediante peticiones REST puras (`fetch`), inyectando el Payload en JSON literal.

66. **Regla 66 (Identidad de Marca - BrandId en Integraciones)**: Al realizar peticiones de firma automatizada (como a la API de DocuSign) u otro servicio a terceros, es de obligado cumplimiento inyectar el identificador visual de marca corporativa (`brandId`), por ejemplo el de *AED Energía*. Su exclusión provocaría el envío de notificaciones y sobres con un diseño neutro ("demo"), perdiendo los colores, logos y alias del remitente (ej. *clientes@aed-energia.com*).

67. **Regla 67 (Rigurosidad de Propiedades Nativas en TypeScript)**: En este CRM, el modelo Prisma `Lead` está altamente normalizado a nivel relacional. No existen columnas como `firstName`, `lastName`, `address`, `city` ni `zipCode` en el Lead (estas propiedades viven en `contractData`/`airtableData` o en `SupplyPoint`). Así mismo, el modelo `Client` no tiene una propiedad `email` o `phone` a secas (sino `contactEmail`, `contactPhone`, `vatNumber` en lugar de `nif`). Cualquier intento de extraer `lead.firstName` o `client.email` romperá el tipado estricto en la compilación de producción. El código debe acceder única y exclusivamente a los campos declarados en `schema.prisma`.

68. **Regla 68 (Deducción Automática del Tipo de Cliente por DNI/NIE)**: El campo `clientType` (Persona Física vs Empresa) NO DEBE requerir una selección manual ni insertarse como "Desconocido". El sistema (tanto en migraciones como en altas nativas) debe inferirlo unívocamente a partir de la primera letra del NIF/CIF (`vatNumber`). Si el documento comienza por cualquier número (0-9) o por las letras X, Y o Z, se clasificará estrictamente como "Persona física" (englobando DNI español y NIE de extranjeros). Cualquier otro prefijo (A, B, E, G, H, J, N, etc., incluyendo las comunidades de propietarios) dictaminará que la entidad es una "Empresa" o Persona Jurídica.

69. **Regla 69 (Propiedad del IBAN y Segregación en Edición de Contratos)**: La pantalla de Modificación de Contrato (ej. `EditContractModal`) **debe omitir deliberadamente** pestañas de edición directa del Cliente o del CUPS. El IBAN de facturación está asociado ineludiblemente a la instalación eléctrica (el Punto de Suministro), y no al Cliente como entidad individual, pues un cliente puede tener varios CUPS con diferentes domiciliaciones bancarias. Por consiguiente, si se requiere modificar un IBAN o dato raíz, se hará navegando a la ficha del Suministro correspondiente, impidiendo que el comercial altere "la foto fija" de un contrato en curso o genere inconsistencias estructurales.

70. **Regla 70 (Automatización de Códigos Técnicos Tipo y Tipo C2)**: Los campos del contrato `tipo` y `tipoC2` (necesarios posteriormente para conformar los XML de la CNMC) jamás deben presentarse como campos modificables o desplegables para el usuario final. El usuario interactuará única y exclusivamente con el selector semántico `tipoTramitacion` (ej: "Alta Nueva", "Cambio comercializadora..."). Es el backend del CRM el responsable absoluto de mapear este concepto comercial y forzar en la base de datos los códigos técnicos `tipo` (ej. 00, 01, 02) y `tipoC2` (ej. 01, 02) correspondientes para evitar discrepancias operativas con las distribuidoras.

71. **Regla 71 (Nomenclatura P1C-P6C vs Consumos Estimados)**: En todas las vistas de detalle y modales de edición del Contrato, la sección de datos energéticos debe denominarse explícitamente "Potencias Contratadas (kW)" para referirse a los campos P1C a P6C. Es un error crítico de UI llamar a estos campos "Estimación de Consumos", ya que la potencia contratada de la instalación (kW) y la previsión anual de consumo (kWh) son conceptos diametralmente opuestos en el sector eléctrico, y la confusión puede provocar problemas serios en la facturación o la tramitación.

72. **Regla 72 (Procesamiento del Webhook de DocuSign y Salvaguarda del PDF Firmado)**: El ciclo de firma digital NO concluye cuando DocuSign emite el evento `envelope-completed` vía webhook. El endpoint del CRM que recepciona el webhook tiene la obligación crítica de: 1) Extraer el `envelopeId` finalizado; 2) Llamar a la API REST de DocuSign (`/documents/combined`) para descargar el PDF consolidado que incrusta el certificado legal de evidencia; 3) Subir físicamente este PDF consolidado a Cloudflare R2; y 4) Registrar la URL permanente resultante en el campo `filePdfSigned` del Contrato en Prisma, marcando el contrato como Aceptado. Sin este paso, el comercial no podrá visualizar ni descargar el contrato final en el sistema.

73. **Regla 73 (Restricción de Consumo y Segregación de SupplyPoint en Contratos)**: El consumo anual estimado (`estimatedMWh`) es la base para el cálculo de comisiones. Durante la generación de un contrato, el CRM debe forzosamente extraer este dato de SIPS (en `sipsRawData.consumo`); si SIPS no lo proporciona, se utilizará el introducido manualmente por el comercial en la edición del Lead. Si la combinación de ambos resulta en 0 o vacío, el sistema abortará la generación del contrato. Adicionalmente, la ventana de edición de un contrato no debe permitir alterar parámetros técnicos del Punto de Suministro (como consumo o dirección); el contrato debe actuar como una foto fija reflejando siempre el `lead.estimatedMWh` con el que fue firmado. Las ediciones del CUPS requerirán una pantalla específica.

74. **Regla 74 (Desglose de Direcciones en SupplyPoint)**: Aunque el esquema de Prisma exige un campo obligatorio `address` (String) para la "Dirección Completa", la interfaz de usuario (UI) **jamás** debe permitir la edición directa de este campo global. La UI debe exponer obligatoriamente los campos desglosados (`Tipo de Vía`, `Nombre Vía`, `Número`, `Piso`, `Puerta` y `Adicional`). Es responsabilidad absoluta del Backend (en los Actions) concatenar estas piezas y reescribir el campo `address` global antes de actualizar la base de datos. Esto garantiza la integridad estricta de los datos estructurados.

75. **Regla 75 (Independencia Visual de Entidades Core e Histórico de CUPS)**: Para evitar el solapamiento de datos y la alteración accidental del registro histórico, las entidades principales (`Client`, `SupplyPoint` y `Contract`) **deben tener siempre sus propias páginas de Vista de Detalle** separadas y aisladas (`/clientes/[id]`, `/puntos-suministro/[id]`, `/contratos/[id]`). Modificar un Punto de Suministro desde el modal de un Contrato está prohibido. Además, la Vista de Detalle del `SupplyPoint` actúa como el "Ledger" (Libro Mayor) físico de una instalación: debe renderizar inexcusablemente una tabla con el Histórico de Contratos (pasados y presentes) que hayan estado vinculados a ese CUPS.

76. **Regla 76 (Ingesta de Facturas en CSV y JSON Dinámico)**: Para la carga masiva de facturas generadas por Ingebau u otros sistemas, el CRM emplea un sistema de ingesta por CSV (`migrate_crm_example.csv`). Para evitar contaminar y sobrecargar el `schema.prisma` con decenas de campos de facturación volátiles (ej. precios P1-P6, lecturas, importes desglosados), la regla de oro es **guardar la fila entera del CSV serializada como un objeto plano dentro del campo JSON `invoiceData`**. En la interfaz de detalle (`InvoiceDetailClient.tsx`), el Front-End lee y mapea estos datos directamente del JSON utilizando funciones seguras como `getRawNum` (capaces de parsear correctamente strings de formato español con comas y puntos). Este diseño garantiza la retrocompatibilidad total: las facturas históricas migradas de Airtable y las nuevas facturas importadas por CSV conviven en el mismo modelo y alimentan la misma UI sin conflictos.

77. **Regla 77 (Mapeo UI Exhaustivo en Vista Detalle de Factura)**: En conjunción con la Regla 76, la vista de detalle de la factura (`InvoiceDetailClient.tsx`) tiene estrictas directrices de renderizado visual para no perder ningún dato vital proveniente de los CSV o del histórico:
    - **Suministro y Periodo**: Debe renderizar explícitamente el `Código Fiscal` (del F1 emitido por distribuidora) y el `Código de Contrato` subyacente de la comercializadora, así como deducir los días facturados restando las fechas `desde` y `hasta`.
    - **Consumos Físicos (Tabla P1-P6)**: Debe desglosar obligatoriamente en formato tabla (grid de 7 columnas) la Energía Activa, Potencia Contratada, Maxímetro, Energía Reactiva y Excedentes Autoconsumo para todos los periodos (P1 a P6).
    - **Lecturas Desde/Hasta**: Se incluirá en la misma tabla de Consumos Físicos una fila específica que muestre las lecturas brutas inicial y final del contador para energía activa (ej: `Lectura Desde AE P1 - Lectura Hasta AE P1`).
    - **Término de Potencia y Energía**: Se deben desglosar todos los Precios Aplicados (`P1P`...`P6P`, `P1E`...`P6E`), las potencias a facturar, y los importes exactos de Peajes ATR y Excesos.
    El objetivo principal es que si el dato existe como columna en el CSV, TIENE que estar expuesto en su correspondiente "SectionCard" de la UI, asegurando que el backoffice tenga la radiografía total de la factura sin necesidad de descargar el PDF o el XML.

78. **Regla 78 (Diccionario de Datos CNMC y Esquemas XSD)**: Para la implementación de flujos de switching nativos (como el proceso **R1 - Reclamaciones**), el CRM depende de la lectura e interpretación de los códigos maestras de la CNMC.
    - Se ha generado un volcado estructurado de las tablas maestras más críticas (ej. Tarifas, Motivos de Rechazo, Motivos de Baja, Tipos de Autoconsumo, etc.) extraídas del documento oficial de la CNMC. Este diccionario vive en el proyecto bajo la ruta: `docs/cnmc_dictionary.json`. El frontend y backend deben consumir este JSON para mapear códigos a descripciones legibles por humanos (ej. traducir tarifa `018` a `2.0TD`, o traducir los tipos de reclamación R1).
    - Adicionalmente, el desarrollo de generadores y parsers XML para los pasos del proceso (ej. `ReclamacionPeticion.xsd`, `AceptacionReclamacion.xsd`, `CierreReclamacion.xsd`) debe construirse obligatoriamente calcando las restricciones de longitud y tipos (types) definidos en los ficheros `.xsd` oficiales ubicados en la carpeta local de red: `Z:\AED\Switching\CNMC - E - V3.0 2024.05.16\CNMC - E - Anexos 2024.05.16\CNMC - E - XSD 2024.05.16\`. El XSD es la autoridad final sobre el esquema de la base de datos para la generación de las reclamaciones.

79. **Regla 79 (Separación Estricta de Apellidos en Base de Datos)**: Airtable combinaba los dos apellidos en un solo campo `APELLIDOS`. Esto es una mala práctica. En el CRM, el esquema de Prisma y los modales (ej. `EditClientModal`) exigen tratar el `lastName` y `lastName2` como datos atómicos separados. En caso de los clientes migrados, un script se encarga de separar el apellido combinado en las columnas nativas. Esto es crítico para poder conformar XMLs oficiales como el de las Reclamaciones o Contratos. Para garantizar la usabilidad, la interfaz debe adaptar sus formularios para personas físicas mostrando los dos campos por separado, pero la "Razón Social" consolidada de la vista general se recalculará de forma opaca al usuario combinando "Nombre + Apellido 1 + Apellido 2" para que las tablas sigan visualizándose bien.

80. **Regla 80 (Identificación y Solución de Clientes Anómalos Airtable)**: Durante la migración, Airtable genera clientes "Desconocidos" si no se introduce un CIF explícito pero sí existen otros campos de datos. También permite introducir erróneamente un DNI de Apoderado como CIF de Empresa, creando dos registros de cliente inconexos para la misma persona. El motor del CRM debe prever y limpiar esta basura, exigiendo el CIF en altas nuevas y permitiendo la edición de la estructura y apellidos en la ficha de cliente para solucionar los datos "legacy" defectuosos.

81. **Regla 81 (Tratamiento Numérico de Energía totalMWh)**: En la tabla `Invoice`, el campo `totalMWh` históricamente importado de Airtable contiene los datos energéticos expresados **en kilovatios hora (kWh)** y no en Megavatios (MWh) como su nombre indica (ej. un valor de `16966` significa 16.966 kWh). En la UI, para que la etiqueta "Energía Consumida" coincida con MWh, **debe dividirse incondicionalmente el valor de BD entre 1000** antes de ser inyectado a formateadores numéricos como `toLocaleString('es-ES')`, forzando 3 decimales (`minimumFractionDigits: 3`). Renderizar directamente el número tal cual resultará en cifras absurdamente abultadas al aplicar el separador de miles español (punto).

82. **Regla 82 (Cálculo del Impuesto Eléctrico por Zona)**:
- El **Impuesto Eléctrico (IE)** no es aplicable o se calcula a tipo 0% en los territorios forales de País Vasco (Guipúzcoa, Vizcaya, Álava) y Navarra para facturas con ciertos límites o regulaciones forales.
- El CRM debe agrupar las Bases Imponibles del Impuesto Eléctrico excluyendo expresamente estas 4 provincias. Estas 4 provincias se agruparán en bloques individuales si se requiere declararlas separadas.
- La base imponible sobre la cual se calcula debe extraerse de `Subtotal 1` (después de descuentos CIE) y no del Total Factura. Las facturas con `taxAmount` (importe impuesto) a 0 **no deben sumar** a la base imponible total declarada.
- **Tipos de Factura**:
  - `Normal`, `Regularización` y `Complementaria` **suman** a la base imponible.
  - `Abono` **resta** a la base imponible.
  - `Rectificativa` **suma** a la base imponible (pues reemplazan a una anulada).
- **Deduplicación**: Para que los importes agregados sean exactos, se debe garantizar que no se suma la misma factura (mismo número) más de una vez, en caso de posibles duplicidades en la subida de ficheros por lotes.

## Regla 83: Unicidad de Facturas por Comercializadora
- Ninguna factura puede estar duplicada dentro de una misma Comercializadora (`Company`).
- **Restricción de Integridad**: El CRM establece una restricción de unicidad a nivel de base de datos (`@@unique([invoiceNumber, companyId])`) para garantizar matemáticamente que un mismo `invoiceNumber` (Número de Factura) jamás exista más de una vez para el mismo código de la comercializadora.
- Para lograr esto, las facturas almacenan directamente una referencia a la Comercializadora (`companyId`) para forzar la validación directa sin necesidad de `JOIN` a través de la marca (`Brand`).

84. **Regla 84 (Búsqueda Difusa y Fallback en Localizaciones Legacy)**: Al buscar facturas por municipio o código postal, es común que la relación `supplyPoint.city` esté vacía o que su nombre no coincida exactamente con la tabla relacional `Municipality` debido a errores tipográficos en los CSVs legacy (ej. "Bollullos del Condado" vs "Bollullos Par del Condado"). El sistema SIEMPRE debe aplicar un algoritmo de búsqueda difusa (fuzzy search) que elimine tildes, ponga en mayúsculas, trocee las cadenas por palabras significativas (>= 4 o 5 letras) y compruebe si comparten palabras clave. Además, si la ciudad relacional es nula, el sistema debe tener un fallback automático para escarbar dentro del JSON `invoiceData` buscando campos nativos como `POBLACION PS` (ignorando explícitamente `POBLACION SOC`, ya que los impuestos municipales gravan el lugar de suministro y no el domicilio del titular).

85. **Regla 85 (Inversión de Signos Numéricos en Facturas de Abono)**: Las facturas de tipo `Abono` importadas históricamente suelen tener sus desgloses económicos internos (`Base Imponible IVA`, `Importe Bono Social`, `Importe Impuesto`, `BaseImponibleF1`) almacenados como valores positivos en el JSON `invoiceData`. Cuando el backend de Prisma extraiga y parsee estos valores para cálculos agregados o para mostrarlos en tablas de detalle (ej. cálculo de Tasa Municipal), **es de obligado cumplimiento invertir su signo a negativo** si se detecta que el tipo de factura incluye la palabra "Abono". Esto garantiza que la suma total sea matemáticamente correcta y que visualmente reflejen un descuento real.

86. **Regla 86 (Unicidad del Punto de Suministro por Comercializadora)**: En la arquitectura multi-comercializadora, el CUPS de un Punto de Suministro (`SupplyPoint`) debe ser único y exclusivo **por Comercializadora (`companyId`)**. Es decir, una misma comercializadora no puede tener dos veces dado de alta el mismo CUPS en su base de datos. Esto garantiza la integridad del historial de contratos y evita expedientes y cruces de facturación duplicados para el mismo emplazamiento.

87. **Regla 87 (Separación de PDFs de Contrato Base y Anexo Técnico)**: Para las modificaciones técnicas de un contrato en vigor (Trámites M1, como cambios de potencia), el proceso de switching de la distribuidora requiere adjuntar un anexo firmado por el cliente. En el modelo relacional `Contract`, el PDF del contrato principal se almacena obligatoriamente separado del anexo, ocupando los campos nativos `filePdfSigned` and `fileAnexoFirmado` respectivamente. Las validaciones de UI (ej. alertas rojas de "Falta Anexo" en Generar Switching) verifican la existencia de este campo (o su homólogo `airtableData['PDF Anexo firmado']` para el histórico de Airtable) exclusivamente cuando el contrato es tipo `M1` y se encuentra en estado `Aceptado`.

88. **Regla 88 (Falsos PDFs en Migración a Cloudflare R2)**: Durante la importación histórica de adjuntos desde Airtable a R2, ciertos documentos subidos por los agentes (fotos de DNI o anexos en JPEG/PNG) heredaron erróneamente la extensión de archivo `.pdf` y un Content-Type `application/pdf`. El CRM asume las URLs proporcionadas por Airtable (ej. `pub-fca..._anexo.pdf`). Si al hacer clic en un enlace de descarga el navegador nativo (Chrome/Edge) arroja el fallo *"Se ha producido un error al cargar el documento PDF"*, no se debe a un error de ruteo del CRM ni de permisos 404/403, sino a un "Falso PDF" (una foto encriptada como PDF). El usuario resolverá este Edge Case mediante descarga ("Guardar enlace como...") y cambio manual de extensión a `.jpg`.

89. **Regla 89 (Modalidades de Subrogación y Persona Jurídica)**: En las modificaciones de titularidad, la interfaz debe adaptarse dinámicamente según la naturaleza del nuevo titular. Para "Persona Física", los apellidos se dividen estrictamente en `Primer Apellido` y `Segundo Apellido`. Para "Persona Jurídica", se usa la `Razón Social` como nombre de la empresa y se obliga a aportar los `Datos del Apoderado` (Nombre, Apellidos y NIF), inyectando estos últimos en las variables de las plantillas PDF de DocuSign (`{{nombrerep}}`, `{{nifrep}}`).

90. **Regla 90 (Clonado de Contratos y Trazabilidad de PDFs)**: En toda subrogación o modificación técnica que genere una nueva versión del contrato, el sistema clonará el contrato base. Es absolutamente crítico que la nueva versión herede íntegramente la URL del contrato original (`filePdfSigned`), de modo que el nuevo titular conserve acceso al acuerdo maestro. El borrador del nuevo anexo se guardará provisionalmente en `pdfUrl` para ser enviado a DocuSign.

91. **Regla 91 (Almacenamiento del Anexo DocuSign)**: Una vez que el webhook de DocuSign reciba la notificación de que el cliente ha firmado el Anexo (detectado por la existencia de `previousContractId`), el documento firmado se insertará directa y exclusivamente en el campo `fileAnexoFirmado`, previniendo así la sobrescritura catastrófica del documento `filePdfSigned` maestro heredado en la Regla 90.

92. **Regla 92 (Gestión Defensiva de CUPS en Subrogaciones)**: Al realizar un cambio de titularidad de un contrato, el sistema nunca debe sobrescribir el punto de suministro (CUPS) original modificando su `clientId` directamente, ya que eso corrompería el historial del antiguo titular. En su lugar, debe comprobar si el nuevo titular ya dispone de ese mismo CUPS en su ficha (mediante `findOrUpdateSupplyPointByCups`). Si no lo posee, se instanciará y creará un CUPS enteramente nuevo para él, garantizando un aislamiento limpio entre los historiales del titular antiguo y el moderno.

93. **Regla 93 (Control de Acceso Granular en UI - Botones de Edición)**: Los botones de acción sensibles que permiten modificar datos maestros del sistema deben renderizarse condicionalmente según el rol del usuario autenticado. Concretamente, los botones **"Editar Punto de Suministro"** y **"Editar Datos"** en las fichas de detalle solo son visibles para los roles `BACKOFFICE`, `COMPANYADMIN` y `SUPERADMIN`. Los roles `COMERCIAL` y `CANAL` solo tienen acceso de **lectura** a estas fichas. Esta regla se aplica en el cliente (`'use client'`) leyendo el rol desde `useSession()` y condicionando el renderizado con un simple `if (role === 'BACKOFFICE' || role === 'COMPANYADMIN' || role === 'SUPERADMIN')`.

94. **Regla 94 (Buscador de Cabecera Global - Deshabilitado por Defecto)**: El componente `<Topbar>` incluye un buscador global en la cabecera (`showSearch` prop). Esta funcionalidad está **deshabilitada por defecto para todas las páginas del sistema** (`showSearch = false` en el componente base). Solo se habilitará de forma explícita y puntual en aquellas páginas que lo requieran mediante `<Topbar showSearch={true} />`. Esta decisión aplica a todos los usuarios y roles sin excepción. No se debe reactivar el buscador en ninguna página nueva sin autorización expresa del cliente.

95. **Regla 95 (Estado FIRMADO de Leads - Cálculo Dinámico)**: El estado "FIRMADO" de un Lead **NO** se obtiene únicamente del campo `status` de la base de datos. Un Lead se considera firmado si cumple **al menos una** de estas condiciones:
    - Su `status` en BD ya es `FIRMADO`.
    - El contrato asociado (`contract`) tiene una `signatureDate` (fecha de firma) informada.
    - El contrato asociado tiene un documento adjunto con tipo `CONTRATO_FIRMADO` en su relación `documents`.
    - Los datos heredados de Airtable (`airtableData`) contienen un campo de PDF firmado.
    Al mostrar estadísticas de Leads (ej. caja verde "Firmados" en el dashboard de Leads), se debe usar este cálculo dinámico y NO un simple `count` del campo `status`. Además, existe un script de corrección masiva (`fix-leads-status.ts`) que sincroniza la BD con esta realidad dinámica, y **debe ejecutarse** cada vez que se sospeche que los contadores están desfasados.

96. **Regla 96 (Sistema de Temas de Marca - Arquitectura Multi-Capa)**: El sistema de personalización visual de marca (colores corporativos, logo) funciona con una arquitectura de tres capas que debe respetarse íntegramente:
    - **Capa 1 - Base de datos**: Los campos `accentColor`, `bgColor`, `surfaceColor`, `borderColor`, `logoUrl`, `faviconUrl` y `name` del modelo `Brand` son la **fuente de verdad permanente**. El Server Action `updateBrandThemeAction` (`src/app/actions/brandThemeActions.ts`) es el único punto autorizado para escribir en estos campos.
    - **Capa 2 - Sesión JWT**: Al iniciar sesión, los colores de la marca primaria del usuario se inyectan en su token. Si el usuario dispone de acceso a varias marcas, el array `allowedBrands` dentro del token contiene los datos completos (id, name, colores) de **todas** las marcas autorizadas. El callback `jwt` en `auth.config.ts` está preparado para recibir el trigger `updateBrandTheme` y actualizar los colores en caliente dentro del token sin obligar al usuario a cerrar sesión.
    - **Capa 3 - BrandSelector + Cookie**: El componente `<BrandSelector>` guarda la marca seleccionada activamente en la cookie `active-brand`. El layout de la aplicación (`src/app/(app)/layout.tsx`) es un **Server Component** que lee esta cookie en cada petición y busca dentro de `allowedBrands` los colores exactos de la marca activa, inyectándolos en el `<BrandThemeProvider>`. Esto garantiza que al cambiar de marca en el selector, toda la interfaz (colores de sidebar, botones, badges) refleje instantáneamente la identidad visual de esa marca sin parpadeos.

97. **Regla 97 (Subida de Logotipo de Marca - Cloudflare R2)**: El logotipo de una marca se gestiona mediante el flujo estándar de subida de archivos del sistema. El formulario de `Ajustes > Marca & Identidad` dispone de un área de drop/click que activa un `<input type="file" hidden>`. Al seleccionar un archivo (PNG, SVG, JPG o WebP, máximo 512 KB), el frontend envía el archivo al endpoint `/api/upload` con el parámetro `folder: 'logos'`. R2 devuelve una URL pública que se guarda temporalmente en el estado del formulario (`logoUrl`). Al pulsar "Guardar cambios", esa URL se persiste junto al resto de datos de la marca en la BD mediante `updateBrandThemeAction`. **Está terminantemente prohibido** guardar el logo en Base64 directamente en PostgreSQL.

98. **Regla 98 (Colores por Defecto del Sistema - Preset SP Energía Lima)**: Cuando se crea una nueva marca o un nuevo usuario sin personalización explícita, el sistema aplica automáticamente la paleta corporativa de SP Energía (el proveedor del software) como fallback. Los valores por defecto definidos en `defaultTheme` de `BrandThemeContext.tsx` son: `accentColor: '#DEFF9A'`, `bgColor: '#0B0F19'`, `surfaceColor: '#111827'`, `borderColor: '#1E2A3A'`. Estos mismos valores son los que el `schema.prisma` asigna por defecto a cada nuevo `Brand`. Este preset aparece en la UI de Ajustes como la primera opción de "Temas predefinidos" bajo el nombre "SP Energía (Lima)" y es el punto de restauración oficial si el cliente quiere deshacer cualquier personalización de colores.

99. **Regla 99 (Visibilidad de Acciones según Rol en Detalle de Contrato y Lead)**: En las páginas de detalle de Contrato y Lead, ciertas acciones administrativas (como editar datos del titular, modificar el punto de suministro, etc.) solo deben mostrarse a roles con privilegios de gestión (`BACKOFFICE`, `COMPANYADMIN`, `SUPERADMIN`). El rol `COMERCIAL` y el rol `CANAL` ven estas páginas en **modo lectura estricto**. Para implementar esta lógica, se extrae el `role` desde `useSession()` en el Client Component correspondiente y se condiciona el renderizado de los botones de edición. No se debe ocultar la información, solo las acciones de modificación.

100. **Regla 100 (Unificación y Estandarización Visual de la UI)**: Para mantener una estética "premium" y consistente en todo el ERP, queda terminantemente prohibido usar cabeceras sueltas o personalizadas por página (ej. un `<div>` con un icono `<h2>`). Todas las páginas maestras del sistema DEBEN importar y utilizar el componente global `<Topbar title="..." subtitle="..." />`. Igualmente, para los contenedores, tablas y tarjetas, no se deben usar clases hardcodeadas de Tailwind como `bg-slate-800 border-slate-700 p-8`. Se deben usar ineludiblemente las clases CSS globales estandarizadas (`.card`, `.data-table`, `.badge-active`, `.badge-draft`) que están programadas en `globals.css` para reaccionar dinámicamente a las variables de color inyectadas por el `BrandThemeProvider` (`var(--bg-elevated)`, `var(--border)`, etc.).

101. **Regla 101 (Jerarquía de Enrutamiento y BrandThemeProvider)**: En la arquitectura de Next.js (App Router) de este proyecto, el proveedor del tema visual multi-marca (`BrandThemeProvider`) y la barra lateral global (`<Sidebar>`) residen exclusivamente dentro del archivo `src/app/(app)/layout.tsx`. Por lo tanto, ninguna página principal (incluido el Dashboard o página raíz) debe colocarse directamente en `src/app/page.tsx`. Toda página que pertenezca a la interfaz del usuario debe estar alojada dentro del grupo de rutas `(app)` (ej. `src/app/(app)/page.tsx`) para asegurar que hereda correctamente el contexto visual y no pierde el logotipo corporativo ni los colores de la marca en la sesión activa.

102. **Regla 102 (Caché Corrupta de Next.js en Alteraciones Estructurales)**: Como ampliación a la Regla 30, no solo los cambios en Prisma pueden romper la caché de Turbopack. Mover archivos estructurales críticos de Next.js (como un `page.tsx` de una carpeta a otra) mientras el servidor de desarrollo (`npm run dev`) está corriendo, provocará inexorablemente errores fatales de compilación (Error 500: Internal Server Error) o errores de TypeScript (`TS2307: Cannot find module`) porque la caché de Next.js sigue apuntando a la ruta fantasma. Para arreglarlo, es obligatorio detener el proceso (Ctrl+C), eliminar la carpeta `.next` (`rm -rf .next` o equivalente en Windows) y volver a ejecutar `npm run dev`.

103. **Regla 103 (Asignación Dinámica de Segmentos y Algoritmo SDA para Grandes Cuentas)**: La etiqueta de segmento de un suministro (ej. `VIP`, `PYME >50 MWh`, `VE >15 MWh`) no es un string fijo caprichoso. El ERP asigna estos segmentos matemáticamente basándose estrictamente en el campo `annualConsumption`, `tariff` y `p1c` del `SupplyPoint`. Para las grandes cuentas (VIPs y Grandes Vehículos Eléctricos), la IA de predicción ignora el Random Forest y utiliza el algoritmo SDA (Similar Day Average), ya que sus consumos son demasiado anómalos o específicos para ser predichos por el árbol de decisión estándar provincial.

104. **Regla 104 (Inconsistencia Estructural del CUPS en Ficheros vs ERP)**: Existe una discrepancia endémica entre los ficheros brutos (como XML/CSVs de distribuidoras en Datadis) y el registro formal del ERP. Las curvas de consumo (`LoadCurve`) suelen traer el CUPS truncado a 20 caracteres (ej. `ES0031104456078001MY`). Sin embargo, en el ERP, los `SupplyPoint` se almacenan habitualmente con el sufijo de control (22 caracteres, ej. `ES0031104456078001MY0F`). Cualquier script o lógica backend que deba emparejar la base maestra de CUPS con curvas importadas DEBE truncar defensivamente los 2 últimos caracteres (si la longitud es 22) para que las consultas y las agregaciones no ignoren el suministro por falsa no-coincidencia.

105. **Regla 105 (Recálculo Selectivo de Históricos para Entornos Serverless)**: El motor predictivo de IA lee de la tabla agrupada `AggregatedLoadCurve` y no de los millones de filas brutas de `LoadCurve`. Sin embargo, en la arquitectura de Vercel, ejecutar una re-agrupación de 365 días provocará un Timeout (Error 504). Para mantener el histórico sincronizado automáticamente con la entrada de nuevos ficheros sin colapsar, el proceso cron debe aplicar un `upsert` o un recalculo **limitado únicamente a los últimos 60 días**. La sincronización profunda de 1 año o superior se reservará exclusivamente para ejecuciones manuales aisladas a través de la UI (Sincronización Profunda) cuando se suban backups completos antiguos.
